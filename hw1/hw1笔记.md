## 1.1 什么是智能体（Agent）

智能体是**通过传感器感知环境，并通过执行器自主采取行动以实现特定目标的实体**。

四个基本要素：**环境、传感器、执行器、自主性**。

### 1.1.1 传统智能体视角

| 类型 | 特点 | 示例 |
|------|------|------|
| **简单反射型** | 基于条件-动作规则，无记忆 | 恒温器 |
| **基于模型的反射型** | 引入内部世界模型，有"记忆" | 隧道中的自动驾驶车 |
| **基于目标型** | 主动选择导向目标的动作 | GPS 导航（A*算法） |
| **基于效用型** | 对状态赋效用值，权衡多目标 | 理性决策系统 |
| **学习型** | 包含性能元件和学习元件 | AlphaGo Zero |

### 1.1.2 LLM 驱动的新范式

传统智能体依赖**显式编程**，LLM 智能体通过海量数据预训练获得**隐式世界模型**。

核心能力：
- **规划与推理**：将高层目标分解为子任务
- **工具使用**：主动调用外部工具填补信息缺口
- **动态修正**：根据反馈调整行为

### 1.1.3 智能体分类

**按时间反应性**：
- **反应式**：低延迟，缺乏远见
- **审议式**：行动前复杂规划（如棋手）
- **混合式**：思考-行动-观察循环

**按知识表示**：
- **符号 AI**：逻辑操作，透明但脆弱
- **亚符号 AI**（连接主义）：神经网络，强大但不透明
- **神经-符号 AI**：融合两者，类比 Kahneman 的系统 1（快思考）和系统 2（慢思考）

> LLM 驱动的智能体是神经-符号 AI 的优秀实践：神经网络核心生成结构化中间步骤（思考、计划、API 调用）。

## 1.2 智能体组成与运行原理

### 1.2.1 任务环境定义（PEAS 模型）

| 要素 | 含义 | 旅行助手示例 |
|------|------|-------------|
| **P** (Performance) | 性能度量 | 推荐准确度、用户满意度 |
| **E** (Environment) | 环境 | 互联网、天气 API、景点数据库 |
| **A** (Actuators) | 执行器 | 调用 API、生成文本 |
| **S** (Sensors) | 传感器 | 接收用户输入、API 返回数据 |

LLM 智能体数字环境的复杂特性：
- **部分可观察**：只能看到 API 返回的数据
- **随机性**：航班价格、库存随时变化
- **多智能体**：其他用户、机器人、动态定价系统
- **序贯且动态**：当前行动影响未来状态

### 1.2.2 智能体运行机制（Agent Loop）

核心机制是 **Agent Loop**（智能体循环）：

```
感知 (Perception) → 思考 (Thought) → 行动 (Action) → 环境状态变化 → 新的感知...
```

1. **感知**：通过传感器/API 接收环境观察
2. **思考**：内部推理
   - 规划：更新理解，分解任务
   - 工具选择：选择合适的工具和参数
3. **行动**：通过执行器执行操作（通常是调用工具）

### 1.2.3 交互协议

每次输出结构：

```
Thought: [自然语言推理]
Action: [具体操作，通常是函数调用]
```

解析器捕获 Action → 执行函数 → 返回 **Observation** → 处理为自然语言进入下一轮循环。

## 1.3 动手实践：5 分钟构建第一个智能体

### 1.3.1 准备工作

依赖包：`requests`、`tavily-python`、`openai`

**系统提示词**定义：
- 角色：旅行助手
- 可用工具：`get_weather`、`get_attraction`
- 输出格式：Thought-Action 对

**工具 1 - get_weather**：调用 `wttr.in` API 获取实时天气

**工具 2 - get_attraction**：使用 Tavily Search API 搜索景点推荐

### 1.3.2 连接 LLM

`OpenAICompatibleClient` 类封装 OpenAI 兼容 API 调用，接收模型名、API Key、Base URL。

### 1.3.3 执行循环

主循环流程：
1. 配置 LLM 客户端
2. 初始化用户提示
3. 最多 5 轮迭代：构建 Prompt → 调用 LLM → 解析 Action → 执行工具 → 记录 Observation

### 1.3.4 运行分析

成功运行示例（3 轮完成）：

| 轮次 | Thought | Action | Observation |
|------|---------|--------|-------------|
| 1 | 需要查询天气 | `get_weather("北京")` | 晴天，26°C |
| 2 | 根据天气推荐景点 | `get_attraction("北京", "Sunny")` | 颐和园、长城 |
| 3 | 信息充足，结束 | `Finish[最终答案]` | 综合推荐 |

体现了四个能力：**任务分解、工具调用、上下文理解、结果综合**。

## 1.4 智能体应用协作模式

### 1.4.1 智能体作为开发者工具

| 工具 | 特点 |
|------|------|
| **GitHub Copilot** | VS Code 集成，实时代码补全 |
| **Claude Code** | 终端助手，理解完整代码库，支持 CI/CD |
| **Trae** | 轻量快速，适合快速原型 |
| **Cursor** | AI 原生编辑器，深度代码理解 |

### 1.4.2 智能体作为自主协作者

架构范式：

1. **单智能体自主循环**（如 AgentGPT）：思考-计划-执行-反思
2. **多智能体协作**：
   - 角色扮演对话（CAMEL）
   - 组织工作流（MetaGPT、CrewAI）
   - 灵活对话网络（AutoGen、AgentScope）
3. **高级控制流**（LangGraph）：状态图建模，支持循环、分支、回溯、人工干预

### 1.4.3 Workflow vs. Agent

| 维度 | Workflow（工作流） | Agent（智能体） |
|------|-------------------|----------------|
| 定义 | 预定义的结构化序列 | 自主追求目标的系统 |
| 决策 | 固定规则和条件 | LLM 动态推理 |
| 示例 | 报销流程（金额→审批链） | 旅行助手（根据天气推理） |
| 灵活性 | 低，流程固定 | 高，适应性强 |

### 题目一：智能体类型分析

请分析以下四个 Case 中的**主体**是否属于智能体，如果是，那么属于哪种类型的智能体（可以从多个分类维度进行分析），并说明理由：

- **Case A**：一台符合冯·诺依曼结构的超级计算机，拥有高达每秒 2EFlop 的峰值算力
- **Case B**：特斯拉自动驾驶系统在高速公路上行驶时，突然检测到前方有障碍物，需要在毫秒级做出刹车或变道决策
- **Case C**：AlphaGo 在与人类棋手对弈时，需要评估当前局面并规划未来数十步的最优策略
- **Case D**：ChatGPT 扮演的智能客服在处理用户投诉时，需要查询订单信息、分析问题原因、提供解决方案并安抚用户情绪

**参考答案：**

| Case | 是否是智能体 | 类型 | 分析 |
|------|-------------|------|------|
| **A** | ❌ 不是 | — | 纯计算工具，没有传感器感知环境，没有自主性，没有目标，只是被动执行指令。算力再强也不等于智能体。 |
| **B** | ✅ 是 | 基于模型的反射型 / 混合型 | 具备完整的感知-决策-行动闭环：摄像头和雷达是**传感器**，刹车和转向是**执行器**，毫秒级自主决策体现**自主性**，目标是安全行驶。能跟踪周围车辆状态（内部模型），同时兼顾反应速度和规划，属于混合型智能体。 |
| **C** | ✅ 是 | 学习型 / 基于目标型 | 感知棋局状态（传感器），自主评估局面并规划未来数十步（自主性），目标是赢得比赛。通过自我对弈不断超越人类策略，体现了学习能力。 |
| **D** | ✅ 是 | 基于 LLM 的智能体 | 感知用户输入（传感器），查询订单、分析问题、生成回复（执行器），自主决策解决方案（自主性），目标是解决用户投诉。具备规划（分解问题）、工具调用（查询订单）、上下文理解等能力。 |

### 题目二：PEAS 模型设计

假设你需要为一个"智能健身教练"设计任务环境。这个智能体能够：

- 通过可穿戴设备监测用户的心率、运动强度等生理数据
- 根据用户的健身目标（减脂/增肌/提升耐力）动态调整训练计划
- 在用户运动过程中提供实时语音指导和动作纠正
- 评估训练效果并给出饮食建议

请使用 PEAS 模型完整描述这个智能体的任务环境，并分析该环境具有哪些特性（如部分可观察、随机性、动态性等）。

**参考答案：**

**PEAS 模型：**

| 要素 | 描述 |
|------|------|
| **P - Performance（性能度量）** | 训练目标达成率（减脂/增肌/耐力是否达标）、动作纠正准确率、饮食建议合理性、用户满意度与坚持率、安全性（避免运动损伤） |
| **E - Environment（环境）** | 用户身体（生理状态、动作姿态）、运动场所（健身房/家中/户外）、饮食环境（食物营养信息）、时间与天气等外部条件 |
| **A - Actuators（执行器）** | 语音输出（耳机/扬声器）— 实时指导、屏幕显示 — 训练计划与数据可视化、推送通知 — 提醒与鼓励、数据存储 — 记录训练历史 |
| **S - Sensors（传感器）** | 可穿戴设备 — 心率/运动强度/GPS、加速度计与陀螺仪 — 动作姿态识别、摄像头 — 动作分析（可选）、用户反馈输入 — 主观感受 |

**环境特性分析：**

| 特性 | 分析 |
|------|------|
| 部分可观察 | 无法完全了解用户的身体状态（如疲劳程度、肌肉酸痛） |
| 随机性 | 生理反应因人而异，同一训练效果不完全确定 |
| 动态性 | 用户状态随时变化（心率波动、疲劳累积） |
| 序贯性 | 今天的训练影响明天的身体状态和训练安排 |
| 多智能体 | 可能与营养师、医生等其他角色协作 |
| 连续性 | 传感器数据是连续流，需要实时处理 |

### 题目三：Workflow vs. Agent 对比分析

某电商公司正在考虑两种方案来处理售后退款申请：

**方案 A（Workflow）**：设计一套固定流程，例如：
- A.1 对于一般商品且在 7 天之内，金额 < 100RMB 自动通过；100-500RMB 由客服审核；>500RMB 需主管审批；而特殊商品（如定制品）一律拒绝退款
- A.2 对于超过 7 天的商品，无论金额，只能由客服审核或主管审批

**方案 B（Agent）**：搭建一个智能体系统，让它理解退款政策、分析用户历史行为、评估商品状况，并自主决策是否批准退款

请分析：
- 这两种方案各自的优缺点是什么？
- 在什么情况下 Workflow 更合适？什么情况下 Agent 更有优势？如果你是该电商公司的负责人，你更倾向于采用哪种方案？
- 是否存在一个方案 C，能够结合两种方案，达到扬长避短的效果？

**参考答案：**

**一、两种方案的优缺点**

| 维度 | 方案 A（Workflow） | 方案 B（Agent） |
|------|-------------------|----------------|
| **优点** | 规则透明，结果可预测；处理速度快（小金额自动通过）；合规性强，便于审计；实现成本低 | 灵活应对复杂场景（如恶意用户识别）；能综合多维信息做判断；可自适应政策变化 |
| **缺点** | 僵硬，难以覆盖所有边界情况（如"定制品但有质量问题"）；规则维护成本随业务复杂度上升；对恶意用户缺乏识别能力 | 决策不可预测，用户缺乏信任感；存在幻觉风险（错误拒绝合理退款）；审计追溯困难；实现和运维成本高 |

**二、适用场景与选择建议**

- **Workflow 更合适**：退款规则明确、业务规模小、合规要求高（如金融、医药行业）
- **Agent 更有优势**：退款场景多样、需要综合判断（用户历史、商品状况）、业务变化频繁
- **作为负责人**：我会选择**方案 C**，纯 Workflow 过于僵硬，纯 Agent 风险过高，混合方案最务实

**三、方案 C（混合方案）**

| 层级 | 处理方式 | 说明 |
|------|---------|------|
| **第一层：规则引擎** | 处理标准场景 | <100RMB 且 7 天内 → 自动通过；定制品且无质量问题 → 自动拒绝。快速覆盖 80% 常规请求 |
| **第二层：智能体审核** | 处理复杂场景 | 规则无法覆盖的请求交给 Agent，综合分析用户历史、商品状况、沟通记录等，给出决策建议 |
| **第三层：人工兜底** | 处理高风险场景 | Agent 置信度低或金额超过阈值时，转人工客服审核，Agent 提供辅助分析报告 |
| **反馈闭环** | 持续优化 | 人工审核结果反馈给 Agent 用于学习，高频边界情况沉淀为新规则补充到第一层 |

> 核心思路：**规则管确定性，Agent 管复杂性，人工管风险性**。让 Agent 在规则无法覆盖的灰色地带发挥作用，同时用人工兜底控制风险。

### 题目四：智能旅行助手功能扩展

在 1.3 节的智能旅行助手基础上，请思考如何添加以下功能（可以只描述设计思路，也可以进一步尝试代码实现）：

> **提示**：思考如何修改 Thought-Action-Observation 循环来实现这些功能。

- 添加一个"记忆"功能，让智能体记住用户的偏好（如喜欢历史文化景点、预算范围等）
- 当推荐的景点门票已售罄时，智能体能够自动推荐备选方案
- 如果用户连续拒绝了 3 个推荐，智能体能够反思并调整推荐策略

**参考答案：**

**功能 1：记忆功能**

在 Agent Loop 外维护一个 `user_profile` 字典，每轮循环注入到 Prompt 中：

```python
user_profile = {
    "preferences": [],      # 用户偏好（历史文化、自然风光等）
    "budget": None,          # 预算范围
    "rejected": [],          # 被拒绝的推荐
    "accepted": []           # 被接受的推荐
}
```

- 新增工具 `save_preference(key, value)`，让 Agent 主动记录用户表达的偏好
- 每轮 Prompt 加入 `user_profile` 内容，让 Agent 拥有跨轮次"记忆"
- System Prompt 中提示："在推荐前先检查用户偏好记录"

**功能 2：门票售罄自动备选**

给 `get_attraction` 加上门票状态检查，售罄时自动触发重新搜索：

- 新增工具 `check_ticket(attraction)` 查询门票状态
- System Prompt 中增加规则："如果景点门票售罄，自动推荐同类替代方案"
- 或直接在 `get_attraction` 内部处理：返回结果附带门票状态，售罄则自动搜索备选

**功能 3：连续拒绝 3 次后反思调整**

在 Agent Loop 中加拒绝计数器，达到阈值时触发反思：

```python
rejection_count = 0

# 每轮检测用户是否拒绝
if user_rejected:
    rejection_count += 1
    if rejection_count >= 3:
        # 注入反思提示到 Prompt
        prompt_history.append("""
        Observation: 用户已连续拒绝3次推荐。请反思：
        1. 之前的推荐有什么问题？
        2. 是否忽略了用户偏好？
        3. 请调整推荐策略，尝试完全不同类型的景点。
        """)
        rejection_count = 0  # 重置计数器
```

> 关键点：反思不是让 Agent 重复之前的逻辑，而是注入一个特殊的 Observation 迫使它改变策略。

### 题目五：神经符号主义的系统 1 与系统 2 协同

卡尼曼的"系统 1"（快速直觉）和"系统 2"（慢速推理）理论为神经符号主义 AI 提供了很好的类比。请首先构思一个具体的智能体的落地应用场景，然后说明场景中的：

> **提示**：医疗诊断助手、法律咨询机器人、金融风控系统等都是常见的应用场景。

- 哪些任务应该由"系统 1"处理？
- 哪些任务应该由"系统 2"处理？
- 这两个系统如何协同工作以达成最终目标？

**参考答案：**

**场景：智能医疗诊断助手**

**系统 1（快速直觉）— 类似神经网络的模式识别：**

| 任务 | 说明 |
|------|------|
| 分诊分级 | 根据症状快速判断紧急程度（胸痛→高优先级） |
| 生命体征监测 | 实时检测心率、血压异常并报警 |
| 常见病识别 | 识别感冒、流感、过敏等常见病症模式 |
| 药物冲突预警 | 即时标记已知的危险药物组合 |
| 医学影像初筛 | 快速识别 X 光/CT 中的明显异常 |

**系统 2（慢速推理）— 类似符号系统的逻辑推演：**

| 任务 | 说明 |
|------|------|
| 复杂鉴别诊断 | 系统分析所有症状，排除不可能的疾病，缩小范围 |
| 治疗方案制定 | 综合病史、过敏、药物冲突、合并症制定方案 |
| 罕见病诊断 | 需要查阅医学文献、分析病例报告 |
| 多因素风险评估 | 结合基因、生活习惯、家族史做综合风险判断 |

**协同机制：**

```
患者就诊
    ↓
系统 1：快速分诊 + 常见病识别
    ↓
├── 常见病且置信度高 → 系统 1 直接输出初步诊断 → 系统 2 复核确认
└── 症状复杂或罕见 → 汇总所有信息交给系统 2
                        ↓
                    系统 2：鉴别诊断 + 文献检索 + 综合推理
                        ↓
                    输出诊断报告 + 置信度 + 建议
```

> 核心思路：系统 1 负责"过滤和快判"，系统 2 负责"深思和决策"。系统 1 低置信度时自动触发系统 2，系统 2 的结论反过来训练系统 1。

### 题目六：LLM 驱动智能体的局限性分析

尽管大语言模型驱动的智能体系统展现出了强大的能力，但它们仍然存在诸多局限。请分析以下问题：

- 为什么智能体或智能体系统有时会产生"幻觉"（生成看似合理但实际错误的信息）？
- 在 1.3 节的案例中，我们设置了最大循环次数为 5 次。如果没有这个限制，智能体可能会陷入什么问题？
- 如何评估一个智能体的"智能"程度？仅使用准确率指标是否足够？

**参考答案：**

**一、为什么会产生"幻觉"？**

源文本指出：亚符号主义系统（神经网络）具有强大的直觉能力，但伴随着不透明性，通常被视为**黑箱（Black Box）**。它能以惊人的准确率识别出图片中的猫，但若问它"为什么你认为这是猫？"，它很可能无法给出合乎逻辑的解释。此外，它在纯粹的逻辑推理任务上表现不佳，有时会产生看似合理却事实错误的幻觉。

具体原因：
- LLM 本质是**概率性文本生成器**，训练目标是"预测下一个最可能的 token"，而非"验证事实正确性"
- 没有内建的事实核查机制，会基于统计模式生成"听起来合理"但错误的内容
- 训练数据可能包含错误或过时信息
- 面对不确定性时，模型倾向于"编造"一个合理答案，而非承认"我不知道"

**二、没有循环次数限制会怎样？**

- **无限循环**：Agent 反复执行相同操作无法收敛（1.3 节案例前 4 轮重复调用 `get_weather` 就是苗头）
- **资源浪费**：持续调用 LLM API 和工具，消耗 token 和费用
- **循环推理**：A 调用 B，B 的结果又触发 A，形成死循环
- **速率限制**：触发 API 限流，导致后续请求全部失败

**三、仅用准确率评估够不够？**

不够，至少还需要多维度评估：

| 维度 | 说明 |
|------|------|
| 任务完成率 | 是否最终完成了用户请求 |
| 效率 | 完成任务用了多少轮循环、多少 token |
| 安全性 | 是否执行了危险操作或泄露敏感信息 |
| 鲁棒性 | 面对异常输入是否能优雅处理 |
| 可解释性 | Thought 部分是否清晰展示了推理过程 |
| 泛化能力 | 对新场景的适应能力 |
| 成本 | API 调用费用是否合理 |
| 用户满意度 | 最终结果是否真正解决了用户问题 |
