## 2.1 基于符号与逻辑的早期智能体

AI 早期探索受数理逻辑影响，研究者相信人类智能可以被形式化符号体系捕捉和复现。这一核心思想催生了**符号主义（Symbolicism）**，也称"逻辑 AI"或"传统 AI"。

### 2.1.1 物理符号系统假说

1976 年，Allen Newell 和 Herbert A. Simon 提出**物理符号系统假说（PSSH）**：

| 论断 | 含义 |
|------|------|
| **充分性** | 任何物理符号系统都具备产生通用智能行为的充分手段 |
| **必要性** | 任何展现通用智能行为的系统，其本质必然是物理符号系统 |

核心主张：智能的本质即符号的计算与处理，将心智研究转化为可工程化实现的问题。

### 2.1.2 专家系统

符号主义时代最重要的应用成果，模拟人类专家在特定领域解决问题的能力。

**核心架构：**

| 组件 | 作用 |
|------|------|
| **知识库** | 存放领域知识，常用**产生式规则**（IF-THEN）表示 |
| **推理机** | 根据事实匹配规则推导结论 |
| **用户界面** | 与用户交互 |

**推理方式：**

- **正向链（Forward Chaining）**：数据驱动，从已知事实出发匹配规则触发结论
- **反向链（Backward Chaining）**：目标驱动，从假设出发寻找能推导出该目标的规则

**MYCIN 系统案例：**

- 斯坦福大学 20 世纪 70 年代开发，辅助诊断细菌性血液感染
- 知识库约 600 条 IF-THEN 规则，主要采用反向链
- 创新性引入**置信因子（CF）**，用 -1 到 1 表示结论可信度
- 评估中表现超过非专业医生，达到人类专家水平

### 2.1.3 SHRDLU

Terry Winograd 于 1968-1970 年开发，在"积木世界"微观环境中通过自然语言与人类交互。

首次将多个 AI 模块集成在统一系统中：

- **自然语言理解**：解析含歧义的英语句子，支持指代消解和上下文记忆
- **规划与行动**：自主规划动作序列完成任务（如先移开障碍物再放积木）
- **记忆与问答**：拥有关于环境和自身行为的记忆

历史地位：综合性智能的典范（感知-思考-行动闭环）、微观世界研究方法的普及、引发关于"符号处理"与"真正理解"之间差异的长期思辨。

### 2.1.4 符号主义面临的根本性挑战

| 挑战 | 说明 |
|------|------|
| **知识获取瓶颈** | 构建知识库成本高昂、耗时漫长，人类许多知识是内隐直觉性的 |
| **常识问题** | 人类行为依赖庞大常识背景，符号系统除非被明确编码否则一无所知（Cyc 项目历经数十年成果仍有限） |
| **框架问题** | 智能体执行动作后如何高效判断哪些事物未发生改变的逻辑难题 |
| **系统脆弱性** | 完全依赖预设规则，遇到规则之外的微小变化可能完全失灵 |

## 2.2 构建基于规则的聊天机器人

### 2.2.1 ELIZA 的设计思想

ELIZA 由 Joseph Weizenbaum 于 1966 年发布，最著名的脚本"DOCTOR"模仿罗杰斯学派非指导性心理治疗师。

工作方式：识别用户输入中的关键词 → 应用预设转换规则 → 将用户陈述转化为开放式提问。

魏泽鲍姆并非要创造真正理解情感的智能体，而是证明**简单句式转换技巧可以在完全不理解内容的情况下营造"智能"和"共情"的假象**。然而许多用户对其产生了情感依赖（**ELIZA 效应**）。

### 2.2.2 模式匹配与文本替换

ELIZA 算法四步骤：

1. **关键词识别与排序**：按优先级选择最高优先级关键词对应规则
2. **分解规则**：用带通配符的分解规则捕获句子其余部分
3. **重组规则**：从关联的重组规则中选择一条生成回应
4. **代词转换**：重组前进行代词转换（如 I→you, my→your）维持对话连贯性

### 2.2.3 规则驱动系统的根本局限性

| 局限 | 说明 |
|------|------|
| 缺乏语义理解 | 不理解词义，如面对"I am not happy"仍机械匹配规则 |
| 无上下文记忆 | 无状态，每次回应仅基于当前单句输入 |
| 规则扩展性问题 | 增加规则导致规模爆炸、冲突与优先级管理复杂 |

> ELIZA 效应揭示了符号主义方法的核心矛盾：看似智能的表现完全依赖预设计规则，面对真实世界语言的无限可能性，穷举式方法注定不可扩展。

## 2.3 马文·明斯基的心智社会

> "The power of intelligence stems from our vast diversity, not from any single, perfect principle."

### 2.3.1 对单一整体智能模型的反思

20 世纪 70-80 年代，符号主义局限性日益明显：专家系统无法拥有儿童般常识；SHRDLU 无法理解积木世界之外的任何事情；ELIZA 对对话内容本身一无所知。

明斯基提出根本性问题：
- "理解"是单一能力还是数十种不同心智过程协同工作的结果？
- "常识"是庞大知识库还是分布式网络？
- 是否应该追求完美统一的逻辑系统？

### 2.3.2 作为协作体的智能

在明斯基框架中：

| 概念 | 说明 |
|------|------|
| **智能体** | 极简单的专门化心智过程，自身是"无心"的（如 LINE-FINDER、GRASP） |
| **机构（Agency）** | 一组协同工作完成更复杂任务的智能体，通过去中心化的激活与抑制信号相互影响 |
| **涌现（Emergence）** | 复杂智能行为从大量简单底层智能体的局部交互中自发产生 |

**搭建积木塔示例：**

```
BUILD-TOWER
    ↓ 激活
BUILDER（循环：只要塔没搭完就激活 ADD-BLOCK）
    ↓ 激活
ADD-BLOCK
    ↓ 依次激活
FIND-BLOCK → GET-BLOCK → PUT-ON-TOP
    ↓ 每个子机构由更底层智能体构成
SEE-SHAPE, REACH, GRASP...
```

没有智能体拥有全局规划，但通过简单激活和抑制规则相互作用，搭建积木塔的行为自然涌现。

### 2.3.3 对多智能体系统的理论启发

心智社会理论为**分布式人工智能（DAI）**和**多智能体系统（MAS）**提供了概念基础：

- **去中心化控制**：不存在中央控制器，设计无中心节点的协调机制
- **涌现式计算**：复杂问题解决方案从简单局部交互规则中自发产生（如蚁群算法、粒子群优化）
- **智能体的社会性**：通信语言（如 ACL）、交互协议（如契约网）、协商策略、信任模型

## 2.4 学习范式的演进与现代智能体

> 如果智能无法被完全设计，是否可以被学习出来？

### 2.4.1 从符号到联结

**联结主义（Connectionism）**在 20 世纪 80 年代重新兴起，灵感来源于生物大脑神经网络：

| 要素 | 说明 |
|------|------|
| 知识的分布式表示 | 知识以连接权重形式分布式存储在大量人工神经元的连接之间 |
| 简单的处理单元 | 每个神经元只执行非常简单的计算 |
| 通过学习调整权重 | 接触大量样本，自动迭代调整连接权重使输出逐渐接近期望目标 |

> 符号主义将人类知识显式编码给机器，联结主义则创造能够像人类一样学习知识的机器。

### 2.4.2 基于强化学习的智能体

**强化学习（RL）**专注于解决序贯决策问题，通过智能体与环境的直接交互在"试错"中学习最大化长期收益。

**核心要素：**

| 要素 | 说明 |
|------|------|
| 智能体（Agent） | 学习者和决策者 |
| 环境（Environment） | 智能体外部的一切 |
| 状态（State, S） | 环境在某一时刻的特定描述 |
| 行动（Action, A） | 根据当前状态能采取的操作 |
| 奖励（Reward, R） | 环境反馈的标量信号 |

**工作模式循环：**

```
观察状态 S_t → 根据策略 π 选择行动 A_t → 执行行动
    ↓
环境转移到新状态 S_{t+1} → 反馈即时奖励 R_{t+1}
    ↓
利用反馈更新优化策略 → 回到观察状态
```

学习目标是最大化**累积奖励（Return）**，而非单一时间步的即时奖励，智能体需要具备"远见"。

**AlphaGo 示例：** 观察棋盘布局（环境状态）→ 决定落子位置（行动）→ 收到胜负信号（奖励）→ 数百万次自我对弈调整策略。

### 2.4.3 基于大规模数据的预训练

RL 智能体缺乏先验知识，需要从零开始。**预训练**在 NLP 领域浮现为解决方案。

**预训练-微调范式：**

1. **预训练阶段**：在互联网级别海量文本上通过**自监督学习**训练超大规模神经网络，学习语言规律、语法结构、事实知识和上下文逻辑，最常见目标是"预测下一个词"
2. **微调阶段**：使用少量特定任务标注数据对模型微调适应对应任务

**LLM 的涌现能力：**

- **上下文学习（In-context Learning）**：无需调整权重，通过 Few-shot 或 Zero-shot 即可完成新任务
- **思维链（Chain-of-Thought）推理**：引导模型输出逐步推理过程，显著提升逻辑和常识推理准确性

> LLM 已演变成兼具海量知识库和通用推理引擎双重角色的组件。

### 2.4.4 基于大语言模型的智能体

LLM 驱动智能体通过多模块协同的持续迭代闭环流程完成任务：

```
感知（Perception）
    ↓ 接收用户指令、API 数据、环境状态变化
思考（Thought）
    ↓ 规划模块分解目标 + LLM 中枢推理决策 + 记忆模块交互
行动（Action）
    ↓ 执行模块解析工具调用指令，调用工具与环境交互
观察与循环
    ↓ 行动结果构成新一轮观察，更新记忆，启动下一轮循环
```

### 2.4.5 智能体发展关键节点概览

三大思潮主导不同时期的研究范式：

| 思潮 | 代表人物 | 关键成果 |
|------|---------|---------|
| **符号主义** | Simon、Minsky | SHRDLU、专家系统、深蓝计算机 |
| **联结主义** | Hinton 等 | 反向传播算法、CNN、Transformer |
| **行为主义** | — | TD-Gammon、AlphaGo |

21 世纪 20 年代，这些思想深度融合：GPT 系列等 LLM 本身是联结主义产物，却成为执行符号推理、工具调用和规划决策的核心"大脑"，形成**神经-符号结合的现代智能体架构**。

## 课后练习

> **提示**：以下部分习题没有标准答案，旨在帮助学习者建立对智能体发展历史的系统性理解，并培养"以史为鉴"的技术洞察力。

### 题目一：物理符号系统假说分析

物理符号系统假说是符号主义时代的理论基石。请分析：

- 该假说的"充分性论断"和"必要性论断"分别是什么含义？
- 结合本章内容，说明符号主义智能体在实践中遇到的哪些问题对该假说的"充分性"提出了挑战？
- 大语言模型驱动的智能体是否符合物理符号系统假说？

**参考答案：**

**一、两个论断的含义**

| 论断 | 含义 | 通俗理解 |
|------|------|---------|
| 充分性 | 任何物理符号系统都具备产生通用智能行为的充分手段 | "能操作符号" → "就能实现智能" |
| 必要性 | 任何展现通用智能行为的系统，其本质必然是物理符号系统 | "有智能" → "一定是符号系统" |

简单说：充分性 = 符号操作是**通往智能的充分路径**，必要性 = 符号操作是**通往智能的唯一路径**。

**二、对"充分性"的挑战**

| 挑战 | 说明 |
|------|------|
| 知识获取瓶颈 | 将人类所有知识显式编码为符号规则，成本极高且不可行 |
| 常识问题 | 人类依赖庞大常识背景，符号系统无法穷举（Cyc 项目数十年仍有限） |
| 框架问题 | 动作执行后判断哪些事物未改变，在逻辑上极其复杂 |
| 系统脆弱性 | 遇到规则之外的微小变化就完全失灵，说明符号操作并不"充分" |

这些实践证明：仅靠符号操作无法产生通用智能，充分性论断被推翻。

**三、LLM 驱动智能体是否符合 PSSH？**

**不符合，至少不完全符合。**

- LLM 的内部机制是**神经网络**（联结主义），不是符号规则系统
- 知识以**连接权重**形式分布式存储，而非显式符号
- LLM 的"推理"本质是**统计模式匹配**，不是逻辑推演

但有趣的是，LLM 驱动智能体的**外部行为**确实涉及符号操作（文本输入/输出、工具调用），Agent Loop 的 Thought-Action-Observation 也是一种符号处理流程。这说明现代智能体是**神经-符号混合系统**，不能简单归类为物理符号系统。

> **结论**：LLM 智能体的存在对 PSSH 的两个论断都构成挑战——它证明智能不需要显式符号规则（挑战必要性），也证明仅靠符号操作不足以实现通用智能（挑战充分性）。

### 题目二：MYCIN 系统反思

专家系统 MYCIN 在医疗诊断领域取得了显著成功，但最终并未大规模应用于临床实践。请思考：

> **提示**：可以从技术、伦理、法律、用户接受度等多个角度分析

- 除了本章提到的"知识获取瓶颈"和"脆弱性"，还有哪些因素可能阻碍了专家系统在医疗等高风险领域的应用？
- 如果让现在的你设计一个医疗诊断智能体，你会如何设计系统来克服 MYCIN 的局限？
- 在哪些垂直领域中，基于规则的专家系统至今仍然是比深度学习更好的选择？请举例说明。

**参考答案：**

**一、MYCIN 未大规模临床应用的其他因素**

| 维度 | 因素 |
|------|------|
| 技术 | 无法实时学习新病原体/菌株；无法与医院电子病历系统（EHR）集成；面对缺失或矛盾数据处理能力有限；推理链过长时解释性反而变差 |
| 伦理 | 系统给出错误建议时责任归属不清（医生？开发者？医院？）；患者是否有权知情并同意 AI 辅助诊断 |
| 法律 | 误诊的法律 liability 问题；医疗法规要求人类医生做最终决策；缺乏医疗器械认证标准 |
| 用户接受度 | 医生担心权威被削弱、不愿依赖"黑箱"建议；患者不信任机器做诊断；医院缺乏部署和维护的专业团队 |

**二、现代医疗诊断智能体设计**

| 模块 | 功能 |
|------|------|
| LLM 中枢 | 理解症状描述、进行鉴别诊断推理 |
| RAG 知识库 | 检索最新医学文献、诊疗指南、药物数据库 |
| 置信度评估 | 低置信度自动转人工审核 |
| EHR 集成 | 自动读取患者病史、过敏、用药记录 |
| 可解释性 | 生成诊断推理链，供医生审查 |
| 反馈闭环 | 医生纠正结果用于持续优化 |

核心原则：**辅助而非替代**，医生始终做最终决策。

**三、规则系统仍优于深度学习的垂直领域**

| 领域 | 原因 |
|------|------|
| 金融合规审查 | 法规明确、规则固定、需要 100% 可解释性 |
| 航空安全检查 | 安全关键、规则确定、不允许概率性判断 |
| 税务计算 | 规则清晰、不允许容错、需要审计追溯 |
| 工业 PLC 控制 | 实时性要求高、规则确定、不能有延迟 |

共同特点：**规则明确、容错率极低、需要完全可解释**。

### 题目三：ELIZA 扩展实践

在 2.2 节中，我们实现了一个简化版的 ELIZA 聊天机器人。请在此基础上进行扩展实践：

> **提示**：这是一道动手实践题，建议实际编写代码

- 为 ELIZA 添加 3-5 条新的规则，使其能够处理更多样化的对话场景（如谈论工作、学习、爱好等）
- 实现一个简单的"上下文记忆"功能：让 ELIZA 能够记住用户在对话中提到的关键信息（如姓名、年龄、职业），并在后续对话中引用
- 对比你扩展后的 ELIZA 与 ChatGPT，列举至少 3 个维度上存在的本质差异
- 为什么基于规则的方法在处理开放域对话时会遇到"组合爆炸"问题并且难以扩展维护？能否使用数学的方法来说明？

**参考答案：**

**第一问和第二问**（添加规则 + 上下文记忆）已在 `hw2.py` 中实现，改动均以 `[新增]` 注释标注。

**三、ELIZA vs ChatGPT 本质差异**

| 维度 | ELIZA | ChatGPT |
|------|-------|---------|
| 语义理解 | 无理解，纯正则模式匹配 | 基于统计的语言理解，能把握语义 |
| 知识储备 | 无知识，仅靠预设规则 | 海量预训练数据，拥有广泛世界知识 |
| 回复生成 | 模板填充，限于预定义响应 | 可生成全新的、上下文相关的回复 |
| 上下文能力 | 无状态（即使加了简单记忆也很有限） | 长上下文窗口，能追踪复杂对话 |
| 学习能力 | 运行时无法学习新模式 | 支持上下文学习（In-context Learning） |
| 开放域覆盖 | 封闭规则集，无法处理未见过的表达 | 几乎能处理任何话题 |
| 推理能力 | 无推理能力 | 能进行思维链（Chain-of-Thought）推理 |

**四、组合爆炸的数学说明**

假设要覆盖 N 个对话主题，每个主题有 M 种不同的表达方式，每个表达需要 K 条规则来处理变体：

- 规则总数 = N × M × K
- 当 N=100, M=10, K=3 时，规则数 = 3,000 条

更根本的问题是，自然语言是**无限生成**的：

- 用户可以说"我今天心情不好"、"我有点难过"、"感觉糟透了"、"生活好累"... 每种意图的表达方式趋近无限
- 规则之间会产生**冲突**：当输入同时匹配多条规则时，优先级管理复杂度随规则数增长
- **维护成本**：修改一条规则可能影响其他规则的匹配结果，回归测试成本随规则数指数增长

用数学表达：设自然语言中表达同一意图的方式集合为 S，|S| → ∞。规则系统试图用有限规则集 R 去覆盖 S，但 |R| << |S|，因此覆盖率 = |R|/|S| → 0。这就是规则方法在开放域对话中注定失败的根本原因。

### 题目四：心智社会理论分析

马文·明斯基在"心智社会"理论中提出了一个革命性的观点：智能源于大量简单智能体的协作，而非单一的完美系统。

- 在"搭建积木塔"的例子中，如果 `GRASP` 智能体突然失效了，整个系统会发生什么？这种去中心化架构的优势和劣势是什么？
- 将"心智社会"理论与现在的一些多智能体系统（如 CAMEL-Workforce、MetaGPT、CrewAI）进行对比，它们之间存在哪些关联和不同之处？
- 马文·明斯基认为智能体可以是"无心"的简单过程，然而现在的大语言模型和智能体往往都拥有强大的推理能力。这是否意味着"心智社会"理论在大语言模型时代不再适用了？

**参考答案：**

**一、GRASP 失效与去中心化架构**

GRASP 失效时，抓取动作无法执行，但 FIND-BLOCK、PUT-ON-TOP 等其他智能体仍正常工作。系统不会像集中式架构那样全面瘫痪，而是**局部功能丧失**。

| 优势 | 劣势 |
|------|------|
| 无单点故障，局部失效不影响全局 | 故障定位困难，难以追踪问题根源 |
| 模块化，易于扩展新智能体 | 全局优化困难，只能达到局部最优 |
| 并行处理，效率高 | 协调开销大，可能产生冲突 |
| 适应性强，可动态重组 | 难以保证行为的确定性和可预测性 |

**二、与现代多智能体系统的对比**

| 维度 | 心智社会 | CAMEL-Workforce | MetaGPT | CrewAI |
|------|---------|-----------------|---------|--------|
| 智能体能力 | "无心"的简单过程 | LLM 驱动，有推理能力 | LLM 驱动，有推理能力 | LLM 驱动，有推理能力 |
| 控制方式 | 完全去中心化 | 角色扮演对话 | 组织化工作流（有 SOP） | 预定义角色和流程 |
| 通信方式 | 激活/抑制信号 | 自然语言对话 | 结构化文档传递 | 自然语言 + 工具调用 |
| 涌现行为 | 核心目标 | 对话中涌现 | 流程驱动，较少涌现 | 流程驱动，较少涌现 |

**关联**：都基于"简单单元协作产生复杂行为"的核心思想，都采用专业化分工。

**不同**：现代 MAS 中每个智能体都有强大推理能力，且往往有中央协调机制，更接近"社会协作"而非"心智涌现"。

**三、LLM 时代是否仍适用？**

**仍然适用，但需要重新理解。**

- 核心洞察（复杂行为从简单交互中涌现）仍然成立
- 但 LLM 改变了基础假设：智能体不再是"无心"的简单过程，而是有强大推理能力的个体
- 这意味着 LLM 时代可能**需要更少的智能体**完成同样的任务，因为每个智能体本身就很强
- 心智社会理论的价值在于**架构思想**（模块化、去中心化、涌现），而非具体的实现方式

> 用一个类比：Minsky 的理论像是"蚂蚁群"模型（简单个体，复杂群体），LLM 时代更像是"专家团队"模型（强大个体，协作放大）。

### 题目五：强化学习理解

强化学习与监督学习是两种不同的学习范式。请分析：

- 用 AlphaGo 的例子说明强化学习的"试错学习"机制是如何工作的
- 为什么强化学习特别适合序贯决策问题？它与监督学习在数据需求上有什么本质区别？
- 现在我们需要训练一个会玩超级马里奥游戏的智能体。如果分别使用监督学习和强化学习，各需要什么数据？哪种方法对于这个任务来说更合适？
- 在大语言模型的训练过程中，强化学习起到了什么关键性的作用？

**参考答案：**

**一、AlphaGo 的试错学习机制**

```
自我对弈循环：
    观察棋盘状态 S_t（当前局面）
        ↓
    选择落子位置 A_t（策略网络 + MCTS 搜索）
        ↓
    对手落子，棋局进入新状态 S_{t+1}
        ↓
    收到奖励 R（最终胜负：+1 赢 / -1 输）
        ↓
    更新策略网络权重，使赢棋概率高的落子被优先选择
        ↓
    重复数百万次自我对弈
```

关键点：没有人类标注的"正确落子"数据，完全通过自我对弈的胜负信号学习策略。

**二、RL 适合序贯决策的原因及与监督学习的区别**

| 维度 | 强化学习 | 监督学习 |
|------|---------|---------|
| 数据来源 | 智能体与环境交互自行产生 | 人工标注的静态数据集 |
| 反馈信号 | 延迟的奖励（整局棋的胜负） | 即时的标签（每步的正确答案） |
| 探索机制 | 主动探索未知策略空间 | 被动接受已知数据 |
| 适合场景 | 序贯决策（当前行动影响未来状态） | 独立预测（输入输出相互独立） |

RL 适合序贯决策的原因：它能学习**长期累积奖励最大化的策略**，而非只优化单步准确率。

**三、超级马里奥智能体的数据需求对比**

| 维度 | 监督学习 | 强化学习 |
|------|---------|---------|
| 需要的数据 | 大量人类高手的游戏录像 + 每帧标注的操作指令 | 仅需要游戏环境本身 |
| 数据获取难度 | 极高：需要海量标注，且不同关卡需要不同数据 | 低：智能体自己玩自己学 |
| 泛化能力 | 差：遇到未见过的场景无法应对 | 好：通过探索学会了通用策略 |
| 适用性 | 不适合（游戏场景无限多） | 非常适合（与环境交互试错） |

**结论**：超级马里奥任务更适合强化学习，因为游戏状态空间无限，无法穷举标注，且需要序贯决策。

**四、RL 在 LLM 训练中的关键作用**

RL 在 LLM 训练中的核心应用是 **RLHF（基于人类反馈的强化学习）**：

```
预训练 LLM → 监督微调（SFT） → RLHF
                                    ↓
                            人类对模型多个回复排序（奖励信号）
                                    ↓
                            用 PPO 等 RL 算法优化模型
                            使模型生成更符合人类偏好的回复
```

关键作用：
- 让 LLM 从"能说话"变成"说好话"——对齐人类偏好
- 解决监督学习无法定义"好回复"的问题（好回复是相对的，需要比较）
- 提升安全性、有用性、诚实性

### 题目六：预训练范式思考

预训练-微调范式是现代人工智能领域的重要突破。请深入思考：

- 为什么说预训练解决了符号主义时代的"知识获取瓶颈"问题？它们在知识表示方式上有什么本质区别？
- 预训练模型的知识绝大部分来自互联网数据，这可能带来哪些问题？如何缓解以上问题？
- 你认为"预训练-微调"范式是否可能会被某种新范式取代？或者它会长期存在？

**参考答案：**

**一、预训练如何解决知识获取瓶颈**

| 维度 | 符号主义 | 预训练 |
|------|---------|--------|
| 知识来源 | 人工从专家处提取，显式编码为规则 | 从海量文本中自动学习，隐式存储在权重中 |
| 获取成本 | 极高：每个领域需要专家 + 知识工程师 | 一次性训练，之后可迁移 |
| 规模 | 受限于人力（MYCIN 的 600 条规则） | 互联网级别（万亿 token） |
| 更新方式 | 手动修改规则 | 重新训练或微调 |
| 知识表示 | 显式符号规则（IF-THEN） | 分布式向量表示（连接权重） |

预训练通过**自监督学习**从海量文本中自动提取知识，将"人告诉机器知识"变成"机器自己学知识"，从根本上解决了知识获取瓶颈。

**二、互联网数据带来的问题及缓解方法**

| 问题 | 说明 | 缓解方法 |
|------|------|---------|
| 偏见与歧视 | 互联网包含大量性别、种族、文化偏见 | 数据去偏、RLHF 对齐、红队测试 |
| 错误信息 | 虚假新闻、阴谋论、过时信息 | RAG 引入可信来源、知识截止日期标注 |
| 隐私泄露 | 训练数据可能包含个人信息 | 数据脱敏、差分隐私、遗忘学习 |
| 版权争议 | 训练数据可能包含受版权保护的内容 | 数据许可管理、来源追溯 |
| 有毒内容 | 互联网包含大量有害内容 | 内容过滤、安全微调、RLHF |

**三、预训练-微调范式是否会被取代？**

**短期内不会被完全取代，但会持续演进。**

可能的演进方向：
- **预训练 + RAG**：不再将所有知识压缩到权重中，而是实时检索外部知识库
- **预训练 + Agent**：模型本身不需要知道一切，通过工具调用获取实时信息
- **持续学习**：模型能够在线更新知识，而非固定在预训练时间点
- **多模态预训练**：从文本扩展到图像、视频、音频的统一预训练

但"在大规模数据上学习通用表示"这一核心思想会长期存在，因为它已经被证明极其有效。

### 题目七：智能代码审查助手设计

假设你要设计一个"智能代码审查助手"，它能够自动审查代码提交（Pull Request），概括代码的实现逻辑、检查代码质量、发现潜在 BUG、提出改进建议。

- 如果在符号主义时代（1980 年代）设计这个系统，你会如何实现？会遇到什么困难？
- 如果在没有大语言模型的深度学习时代（2015 年左右），你会如何实现？
- 在当前的大语言模型和智能体的时代，你会如何设计这个智能体的架构？它应该包含哪些模块（参考图 2.10）？
- 对比这三个时代的方案，说明智能体技术的演进如何使这个任务从"几乎不可能"变为"可行"

**参考答案：**

**一、符号主义时代（1980 年代）**

**实现方式**：基于规则的静态代码分析工具（如 Lint）。

- 编写大量 IF-THEN 规则检测常见问题（未声明变量、类型不匹配、空指针等）
- 使用抽象语法树（AST）解析代码结构

**困难**：
- 规则只能覆盖已知模式，无法理解代码的**业务逻辑**
- 无法概括代码的"实现逻辑"——需要理解自然语言注释和变量命名
- 规则维护成本高，新语言/框架需要重新编写规则
- 无法提出"改进建议"——因为不理解代码的意图

**结论**：能做语法检查，但几乎不可能做到"理解代码逻辑"和"提出改进建议"。

**二、深度学习时代（2015 年左右）**

**实现方式**：基于 NLP 模型的代码分析。

- 使用 Word2Vec/GloVe 将代码 token 化为向量
- 用 CNN/RNN 模型训练代码分类器（如检测 bug 模式、代码异味）
- 使用序列到序列模型生成代码摘要

**困难**：
- 需要大量标注数据（代码 + 对应的审查意见），获取成本极高
- 模型泛化能力差：在 Java 上训练的模型无法用于 Python
- 无法理解跨文件的代码关系和项目级上下文
- 生成的摘要质量低，缺乏可解释性

**结论**：能做到简单的 bug 检测和代码分类，但距离"智能审查"差距很大。

**三、LLM 和智能体时代（当前）**

**架构设计**（参考图 2.10 的智能体组件）：

| 模块 | 功能 |
|------|------|
| **感知模块** | 读取 PR 的 diff、文件内容、commit message、项目文档 |
| **LLM 中枢** | 理解代码逻辑、推理潜在问题、生成自然语言审查意见 |
| **记忆模块** | 存储项目历史审查记录、编码规范、常见问题模式 |
| **工具箱** | 调用静态分析工具（ESLint）、运行测试套件、查询文档、搜索相似代码 |
| **规划模块** | 决定审查策略：先检查安全性 → 再检查逻辑 → 最后检查风格 |
| **执行模块** | 生成结构化审查报告，标注问题位置和严重级别 |

**Agent Loop**：
1. 感知：读取 PR diff 和项目上下文
2. 思考：LLM 分析代码逻辑，规划审查步骤
3. 行动：调用静态分析工具、运行测试、搜索类似问题
4. 观察：收集工具返回结果
5. 循环：综合所有信息生成最终审查报告

**四、三个时代对比**

| 维度 | 符号主义（1980s） | 深度学习（2015） | LLM Agent（当前） |
|------|------------------|-----------------|-------------------|
| 理解能力 | 语法级 | 模式级 | 语义级 |
| 跨文件分析 | 不支持 | 有限 | 支持（长上下文） |
| 生成能力 | 无 | 有限（模板化） | 强（自然语言） |
| 可扩展性 | 差（需手动写规则） | 中（需标注数据） | 强（提示工程即可） |
| 任务完成度 | 检查语法 | 检测 bug 模式 | 理解逻辑 + 提建议 |

> 演进路径：从"检查语法"到"检测模式"到"理解语义"。LLM 的出现使代码审查从"规则匹配"升级为"智能理解"，任务从"几乎不可能"变为"可行"。
