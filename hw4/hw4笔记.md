## 4.2 ReAct（Reasoning and Acting）

### 4.2.1 ReAct 的工作流程

ReAct 由 Shunyu Yao 于 2022 年提出，核心思想是将**推理（Reasoning）**与**行动（Acting）**显式结合，形成"思考-行动-观察"循环。

在 ReAct 之前，主流方法分两类：
- **纯思考**（如 CoT）：能推理但无法与外部世界交互，易产生幻觉
- **纯行动**：直接输出动作，但缺乏规划和纠错能力

ReAct 的每一步遵循固定轨迹：

```
Thought（思考）→ Action（行动）→ Observation（观察）→ 循环...
```

形式化表达：在每个时间步 t，智能体策略 π 根据问题 q 和历史轨迹生成思考和行动：

$$(th_t, a_t) = \pi(q, (a_1, o_1), \ldots, (a_{t-1}, o_{t-1}))$$

工具 T 执行行动并返回观察：

$$o_t = T(a_t)$$

**适用场景：**
- 需要外部知识的任务（实时信息、专业知识）
- 需要精确计算的任务（交给计算器工具）
- 需要与 API 交互的任务（数据库、服务调用）

### 4.2.2 工具的定义与实现

一个良好定义的工具包含三个核心要素：

| 要素 | 说明 |
|------|------|
| 名称（Name） | 简洁唯一的标识符，如 `Search` |
| 描述（Description） | 清晰的自然语言描述，LLM 依赖它判断何时使用 |
| 执行逻辑（Execution Logic） | 真正执行任务的函数 |

**ToolExecutor 类**提供统一的工具管理：

| 方法 | 功能 |
|------|------|
| `registerTool` | 注册新工具（名称 + 描述 + 函数） |
| `getTool` | 根据名称获取工具函数 |
| `getAvailableTools` | 获取所有工具的格式化描述 |

### 4.2.3 ReAct 智能体的编码实现

**核心组件：**

| 组件 | 作用 |
|------|------|
| 系统提示词 | 定义角色、工具清单、输出格式、动态上下文 |
| 核心循环 | 格式化提示词 → 调用 LLM → 执行动作 → 整合结果 |
| 输出解析器 | 用正则表达式从 LLM 输出中提取 Thought 和 Action |
| 工具调用 | 检查 Finish 指令 → 获取工具函数 → 执行 → 返回 Observation |
| 历史整合 | 将 Action 和 Observation 追加到 history，形成闭环 |

**ReAct 提示词模板核心结构：**

```
角色定义 + 工具清单 + 格式规约（Thought/Action） + 动态上下文（question/history）
```

**运行流程：**

```
while current_step < max_steps:
    1. 格式化提示词（注入工具描述 + 历史记录）
    2. 调用 LLM 生成 Thought + Action
    3. 解析输出
    4. 如果 Action 是 Finish → 提取答案，结束
    5. 否则执行工具，获得 Observation
    6. 将 Action + Observation 追加到历史
```

### 4.2.4 ReAct 的特点、局限性与调试技巧

**主要特点：**

| 特点 | 说明 |
|------|------|
| 高可解释性 | Thought 链清晰展示每一步的推理过程 |
| 动态规划与纠错 | 根据 Observation 动态调整后续行动 |
| 工具协同 | LLM 负责推理，工具负责执行，突破单一 LLM 局限 |

**固有局限性：**

| 局限 | 说明 |
|------|------|
| 强依赖 LLM 能力 | 推理能力不足会导致错误规划或格式错误 |
| 执行效率低 | 串行循环，多次 LLM 调用带来延迟和成本 |
| 提示词脆弱 | 模板微小变动可能影响 LLM 行为 |
| 可能陷入局部最优 | 步进式决策缺乏全局规划，可能原地打转 |

**调试技巧：**
- 检查完整提示词（追溯 LLM 决策源头）
- 分析原始输出（判断是格式问题还是解析问题）
- 验证工具输入输出格式
- 调整提示词中的 Few-shot 示例
- 尝试不同模型或参数（temperature 通常设为 0）

## 4.3 Plan-and-Solve

### 4.3.1 Plan-and-Solve 的工作原理

Plan-and-Solve 由 Lei Wang 在 2023 年提出，核心动机是解决 CoT 在处理多步骤复杂问题时容易"偏离轨道"的问题。

与 ReAct 的区别：
- **ReAct**：边想边做，走一步看一步（像侦探）
- **Plan-and-Solve**：先规划完整蓝图，再严格执行（像建筑师）

两阶段流程：

| 阶段 | 说明 |
|------|------|
| **规划阶段（Planning）** | 接收完整问题，分解为分步骤的行动计划 |
| **执行阶段（Solving）** | 严格按照计划逐一执行，每步可能是一次 LLM 调用 |

形式化表达：

$$P = \pi_{plan}(q) = (p_1, p_2, \ldots, p_n)$$

$$s_i = \pi_{solve}(q, P, (s_1, \ldots, s_{i-1}))$$

**适用场景：**
- 多步数学应用题
- 需要整合多个信息源的报告撰写
- 代码生成任务（先构思结构再实现）

### 4.3.2 规划阶段

**Planner 提示词设计要点：**

| 要点 | 说明 |
|------|------|
| 角色设定 | "顶级的 AI 规划专家"，激发模型专业能力 |
| 任务描述 | 清晰定义"分解问题"的目标 |
| 格式约束 | 强制输出为 Python 列表格式，简化后续解析 |

**Planner 类核心逻辑：**
1. 格式化提示词
2. 调用 LLM 生成计划
3. 用 `ast.literal_eval` 安全解析 Python 列表字符串
4. 返回步骤列表

### 4.3.3 执行器与状态管理

**Executor 提示词包含四部分：**

| 部分 | 说明 |
|------|------|
| 原始问题 | 确保模型始终了解最终目标 |
| 完整计划 | 让模型了解当前步骤在整体中的位置 |
| 历史步骤与结果 | 提供至今已完成的工作作为上下文 |
| 当前步骤 | 明确指示现在需要解决的具体任务 |

**Executor 类核心逻辑：**
1. 循环遍历计划中的每个步骤
2. 格式化提示词（注入历史）
3. 调用 LLM 解决当前步骤
4. 更新历史记录，为下一步做准备

**PlanAndSolveAgent** 作为协调者，组合 Planner 和 Executor。

### 4.3.4 运行实例分析

以数学应用题为例：
- 规划阶段成功分解为 4 个逻辑步骤
- 执行阶段严格按计划逐步计算
- 每步正确使用上一步的结果
- 最终得出正确答案

## 4.4 Reflection

### 4.4.1 Reflection 机制的核心思想

Reflection 灵感来源于人类学习过程：完成初稿后校对，解题后验算。核心工作流：**执行 → 反思 → 优化**。

| 阶段 | 说明 |
|------|------|
| 执行（Execution） | 用 ReAct 或 Plan-and-Solve 生成初步方案（"初稿"） |
| 反思（Reflection） | 调用 LLM 扮演"评审员"，评估事实性、逻辑性、效率、遗漏 |
| 优化（Refinement） | 将"初稿"和"反馈"作为上下文，生成"修订稿" |

形式化表达：

$$F_i = \pi_{reflect}(Task, O_i)$$

$$O_{i+1} = \pi_{refine}(Task, O_i, F_i)$$

**Reflection 的价值：**
- 提供内部纠错回路，修正高层次逻辑和策略错误
- 将一次性任务转变为持续优化过程
- 构建临时"短期记忆"，记录从初稿到最终版的迭代轨迹

### 4.4.2 案例设定与记忆模块设计

**案例任务**：编写 Python 函数找出 1 到 n 之间所有素数。

选择原因：
- 存在明确的优化路径（简单递归 → 高效筛法）
- 反思点清晰（时间复杂度过高）
- 优化方向明确（埃拉托斯特尼筛法）

**Memory 类**：存储每次"执行-反思"循环的完整轨迹。

| 方法 | 功能 |
|------|------|
| `add_record` | 添加记录（类型：execution/reflection） |
| `get_trajectory` | 将所有记录格式化为连贯文本，用于构建提示词 |
| `get_last_execution` | 获取最近一次的执行结果 |

### 4.4.3 Reflection 智能体的编码实现

**三个提示词模板：**

| 模板 | 角色 | 作用 |
|------|------|------|
| 初始执行提示词 | 资深 Python 程序员 | 首次尝试解决问题 |
| 反思提示词 | 极其严格的代码评审专家 | 审查代码，找出算法效率瓶颈 |
| 优化提示词 | 资深 Python 程序员 | 根据反馈修正和优化代码 |

**ReflectionAgent 核心流程：**

```
1. 初始执行 → 生成代码 → 存入记忆
2. for i in range(max_iterations):
    a. 反思 → 生成反馈 → 存入记忆
    b. 如果反馈包含"无需改进" → 结束
    c. 优化 → 生成新代码 → 存入记忆
3. 返回最终代码
```

### 4.4.4 运行实例分析

以素数函数为例：
- **初始版本**：O(n × √n) 的试除法
- **第 1 轮反思**：精准指出时间复杂度瓶颈，建议使用筛法
- **第 1 轮优化**：实现埃拉托斯特尼筛法，降至 O(n log log n)
- **第 2 轮反思**：肯定当前效率，提及更高级优化但判断"无需改进"
- **终止**：反思认为代码已无需改进

### 4.4.5 Reflection 的成本收益分析

| 成本 | 说明 |
|------|------|
| 调用开销增加 | 每轮迭代至少额外调用两次 LLM |
| 延迟显著提高 | 串行过程，总耗时成倍延长 |
| 提示工程复杂度上升 | 不同阶段需要不同角色的提示词 |

| 收益 | 说明 |
|------|------|
| 质量跃迁 | 从"合格"迭代优化为"优秀" |
| 鲁棒性增强 | 发现并修复逻辑漏洞、事实错误、边界情况 |

> 典型的"以成本换质量"策略，适合对结果质量要求极高、对实时性要求宽松的场景。

## 本章小结

| 范式 | 核心思想 | 优势 | 适用场景 |
|------|---------|------|---------|
| **ReAct** | 思考-行动-观察循环 | 高可解释性、动态纠错、工具协同 | 探索性任务、需要外部工具输入 |
| **Plan-and-Solve** | 先规划后执行 | 结构性强、目标一致 | 逻辑路径确定、内部推理密集 |
| **Reflection** | 执行-反思-优化迭代 | 质量跃迁、鲁棒性强 | 对结果准确性和可靠性要求极高 |

**选择策略：**

| 任务特征 | 推荐范式 |
|---------|---------|
| 需要探索和外部信息 | ReAct |
| 任务结构清晰可分解 | Plan-and-Solve |
| 对结果质量要求极高 | Reflection |
| 复杂任务 | 混合范式（如 Plan + ReAct + Reflection） |

### 题目一：三种范式对比分析

本章介绍了三种经典的智能体范式：ReAct、Plan-and-Solve 和 Reflection。请分析：

- 这三种范式在"思考"与"行动"的组织方式上有什么本质区别？
- 如果要设计一个"智能家居控制助手"（需要控制灯光、空调、窗帘等多个设备，并根据用户习惯自动调节），你会选择哪种范式作为基础架构？为什么？
- 是否可以将这三种范式进行组合使用？若可以，请尝试设计一个混合范式的智能体架构，并说明其适用场景。

**参考答案：**

**一、本质区别**

| 维度 | ReAct | Plan-and-Solve | Reflection |
|------|-------|----------------|------------|
| 思考方式 | 每步都思考，边想边做 | 一次性想清楚，再动手 | 做完后再想，想完再改 |
| 行动方式 | 思考与行动交替进行 | 规划完成后严格按计划执行 | 先执行，再根据反思修正 |
| 信息流 | Thought ↔ Action ↔ Observation | Plan → Step1 → Step2 → ... | Execution → Reflection → Refinement |
| 适应性 | 动态调整，每步可改方向 | 静态计划，执行中不改方向 | 迭代优化，每轮可改方案 |

**二、智能家居控制助手**

推荐使用 **ReAct** 作为基础架构，原因：
- 智能家居是**实时交互场景**，需要根据传感器反馈动态调整（如温度到了就停止空调）
- 需要**调用外部工具**（控制不同设备的 API）
- 用户指令可能是模糊的（"有点热"），需要根据 Observation（当前温度）动态决策

如果需要更复杂的场景（如"准备一个浪漫晚餐"），可以在 ReAct 基础上加入 Plan-and-Solve 的规划能力。

**三、混合范式设计**

```
用户输入
    ↓
Plan-and-Solve：生成整体计划（调暗灯光 → 播放音乐 → 调节温度）
    ↓
ReAct：逐步执行每个计划步骤（思考 → 调用工具 → 观察结果）
    ↓
Reflection：执行完成后反思（效果如何？是否需要调整？）
    ↓
输出优化后的结果
```

适用场景：复杂的多步骤任务，如"安排一次家庭聚会"（需要规划场地、食物、设备控制，每步需要实时交互，最终需要反思优化）。

### 题目二：输出解析方案改进

在 4.2 节的 ReAct 实现中，我们使用了正则表达式来解析大语言模型的输出（如 Thought 和 Action）。请思考：

- 当前的解析方法存在哪些潜在的脆弱性？在什么情况下可能会失败？
- 除了正则表达式，还有哪些更鲁棒的输出解析方案？
- 尝试修改本章的代码，使用一种更可靠的输出格式，并对比两种方案的优缺点

**参考答案：**

**一、正则表达式的脆弱性**

| 场景 | 问题 |
|------|------|
| LLM 输出格式微调 | "Thought：" vs "Thought:" vs "思考："，冒号全角半角不同 |
| 多行 Thought | 正则可能在第一个 Action 处截断，丢失后续内容 |
| Action 中包含方括号 | `Search[北京[朝阳区]]` 会导致正则匹配错误 |
| LLM 输出额外内容 | 开头或结尾的解释文字可能干扰匹配 |
| 模型不遵循格式 | 能力较弱的模型可能完全不按模板输出 |

**二、更鲁棒的解析方案**

| 方案 | 说明 |
|------|------|
| JSON 格式输出 | 要求 LLM 输出 `{"thought": "...", "action": "..."}`，用 `json.loads` 解析 |
| 结构化标记 | 使用特殊分隔符如 `###THOUGHT###` 和 `###ACTION###` |
| Function Calling | 使用 OpenAI 的 function calling 接口，模型直接输出结构化参数 |
| Pydantic 模型 | 定义输出 schema，用 `instructor` 库强制模型按 schema 输出 |

**三、方案对比**

| 维度 | 正则表达式 | JSON 格式 | Function Calling |
|------|-----------|----------|-----------------|
| 实现难度 | 低 | 中 | 低 |
| 鲁棒性 | 差 | 好 | 最好 |
| 兼容性 | 所有模型 | 所有模型 | 仅支持 FC 的模型 |
| 灵活性 | 高 | 中 | 低（受 schema 约束） |

> 建议：优先使用 Function Calling（如果模型支持），其次用 JSON 格式，正则表达式仅作为兜底方案。

### 题目三：工具扩展实践

> **提示**：这是一道动手实践题，建议实际编写代码

基于 4.2.2 节的 ToolExecutor 设计，请完成以下扩展实践：

- 为 ReAct 智能体添加一个"计算器"工具，使其能够处理复杂的数学计算问题（如"计算 (123 + 456) × 789 / 12 = ? 的结果"）
- 设计并实现一个"工具选择失败"的处理机制：当智能体多次调用错误的工具或提供错误的参数时，系统应该如何引导它纠正？
- 思考：如果可调用工具的数量增加到 50 个甚至 100 个，当前的工具描述方式是否还能有效工作？在可调用工具数量随业务需求显著增加时，从工程角度如何优化工具的组织和检索机制？

**参考答案：**

**一、计算器工具实现**

```python
def calculator(expression: str) -> str:
    """安全地计算数学表达式"""
    import ast
    import operator
    allowed_ops = {
        ast.Add: operator.add, ast.Sub: operator.sub,
        ast.Mult: operator.mul, ast.Div: operator.truediv,
        ast.Pow: operator.pow, ast.Mod: operator.mod,
    }
    try:
        tree = ast.parse(expression, mode='eval')
        # 安全检查：只允许数字和允许的运算符
        for node in ast.walk(tree):
            if isinstance(node, (ast.Expression, ast.BinOp, ast.Constant, ast.UnaryOp)):
                continue
            return f"错误: 不允许的操作 {type(node).__name__}"
        result = eval(compile(tree, '<string>', 'eval'))
        return str(result)
    except Exception as e:
        return f"计算错误: {e}"
```

**二、工具选择失败处理机制**

```
错误计数器 = 0
最大重试 = 3

while 错误计数器 < 最大重试:
    如果工具调用失败:
        错误计数器 += 1
        将错误信息加入 Observation（如"工具不存在"或"参数格式错误"）
        在下一轮 Thought 中，LLM 会看到错误并尝试修正
    如果连续 3 次失败:
        注入特殊提示："你已多次调用失败，请重新分析问题，考虑是否需要使用其他工具或直接回答"
```

**三、大规模工具优化**

当工具数量达到 50-100 个时，将所有工具描述塞入提示词会导致：
- Token 消耗过大
- LLM 选择困难（信息过载）

优化方案：

| 方案 | 说明 |
|------|------|
| **工具分类分层** | 先选类别（搜索类/计算类/数据库类），再选具体工具 |
| **向量检索** | 将工具描述向量化，根据用户问题检索 Top-K 相关工具 |
| **动态工具注册** | 按需加载工具，不把所有工具都塞进提示词 |
| **工具摘要** | 先展示简短摘要，LLM 选择后再展示详细参数 |

### 题目四：Plan-and-Solve 深入分析

Plan-and-Solve 范式将任务分解为"规划"和"执行"两个阶段。请深入分析：

- 在 4.3 节的实现中，规划阶段生成的计划是"静态"的（一次性生成，不可修改）。如果在执行过程中发现某个步骤无法完成或结果不符合预期，应该如何设计一个"动态重规划"机制？
- 对比 Plan-and-Solve 与 ReAct：在处理"预订一次从北京到上海的商务旅行（包括机票、酒店、租车）"这样的任务时，哪种范式更合适？为什么？
- 尝试设计一个"分层规划"系统：先生成高层次的抽象计划，然后针对每个高层步骤再生成详细的子计划。这种设计有什么优势？

**参考答案：**

**一、动态重规划机制**

```
while 有未完成的步骤:
    执行当前步骤
    如果执行失败或结果不符合预期:
        将失败原因和已有结果作为上下文
        调用 Planner 重新生成剩余步骤的计划
        继续执行新计划
```

关键点：不是重新生成整个计划，而是**从失败点开始重规划**，保留已完成步骤的结果。

**二、旅行预订任务对比**

推荐使用 **Plan-and-Solve + ReAct 混合**：

| 阶段 | 范式 | 原因 |
|------|------|------|
| 整体规划 | Plan-and-Solve | 旅行涉及多个子任务，需要先规划好顺序（先订机票再订酒店） |
| 每步执行 | ReAct | 每个子任务需要与外部 API 交互（查询航班、比价、预订），需要动态调整 |

纯 ReAct 的问题：缺乏全局规划，可能先订了酒店再发现没有合适航班。
纯 Plan-and-Solve 的问题：机票售罄时无法动态调整后续计划。

**三、分层规划系统**

```
高层规划（Plan-and-Solve）:
    ["预订交通", "预订住宿", "安排活动"]

每个高层步骤的子规划（ReAct 或嵌套 Plan-and-Solve）:
    "预订交通" → ["查询航班", "比较价格", "预订机票", "确认订单"]
    "预订住宿" → ["查询酒店", "比较评分", "预订房间"]
```

优势：
- **降低规划复杂度**：每层只需要关注当前粒度的任务
- **提高灵活性**：某一层失败不影响其他层的规划
- **支持并行**：不同高层步骤可以并行执行

### 题目五：Reflection 机制思考

Reflection 机制通过"执行-反思-优化"循环来提升输出质量。请思考：

- 在 4.4 节的代码生成案例中，不同阶段使用的是同一个模型。如果使用两个不同的模型（例如，用一个更强大的模型来做反思，用一个更快的模型来做执行），会带来什么影响？
- Reflection 机制的终止条件是"反馈中包含无需改进"或"达到最大迭代次数"。这种设计是否合理？能否设计一个更智能的终止条件？
- 假设你要搭建一个"学术论文写作助手"，它能够生成初稿并不断优化论文内容。请设计一个多维度的 Reflection 机制，从段落逻辑性、方法创新性、语言表达、引用规范等多个角度进行反思和改进。

**参考答案：**

**一、使用不同模型的影响**

| 维度 | 同一模型 | 不同模型（强反思 + 快执行） |
|------|---------|--------------------------|
| 质量 | 反思能力受限于模型本身 | 强模型能发现更深层次的问题 |
| 成本 | 较低 | 较高（强模型 API 更贵） |
| 速度 | 一致 | 快模型执行快，强模型反思慢 |
| 一致性 | 高（同一"思维模式"） | 可能出现"理解偏差"（两个模型对同一代码的理解不同） |

建议：执行阶段用快模型（成本低），反思阶段用强模型（质量高），但需要确保两个模型对任务的理解一致。

**二、更智能的终止条件**

当前设计的问题："无需改进"可能被模型轻易说出（模型倾向于同意自己）。

改进方案：

| 条件 | 说明 |
|------|------|
| 量化评分 | 让反思模型对代码打分（1-10），连续两轮评分差距 < 0.5 时终止 |
| 外部验证 | 运行测试用例，通过率不再提升时终止 |
| 多维度检查 | 分别评估正确性、效率、可读性，全部达标时终止 |
| 成本预算 | 设定最大 token 消耗，达到预算时强制终止 |

**三、论文写作助手的多维度 Reflection**

```
初始执行：生成论文初稿

反思阶段（多维度并行评审）：
    评审员1（逻辑专家）：检查段落逻辑性、论证连贯性
    评审员2（领域专家）：评估方法创新性、实验设计合理性
    评审员3（语言专家）：检查语言表达、语法、学术规范
    评审员4（引用专家）：验证引用规范、参考文献完整性

优化阶段：
    汇总所有评审反馈
    按优先级修复问题（逻辑 > 创新性 > 语言 > 引用）
    生成修订稿

循环直到所有维度评分达标
```

### 题目六：提示词工程分析

提示词工程是影响智能体最终效果的关键技术。本章展示了多个精心设计的提示词模板。请分析：

- 对比 4.2.3 节的 ReAct 提示词和 4.3.2 节的 Plan-and-Solve 提示词，它们显然存在结构设计上的明显不同，这些差异是如何服务于各自范式的核心逻辑的？
- 在 4.4.3 节的 Reflection 提示词中，我们使用了"你是一位极其严格的代码评审专家"这样的角色设定。尝试修改这个角色设定（如改为"你是一位注重代码可读性的开源项目维护者"），观察输出结果的变化，并总结角色设定对智能体行为的影响。
- 在提示词中加入 few-shot 示例往往能显著提升模型对特定格式的遵循能力。请为本章的某个智能体尝试添加 few-shot 示例，并对比其效果。

**参考答案：**

**一、ReAct vs Plan-and-Solve 提示词差异**

| 维度 | ReAct 提示词 | Plan-and-Solve 提示词 |
|------|-------------|----------------------|
| 核心指令 | 每步输出 Thought + Action | 一次性输出完整计划列表 |
| 格式要求 | 自由文本 + Action 格式 | 严格的 Python 列表格式 |
| 上下文 | 注入历史（每步累积） | 仅注入原始问题 |
| 设计目的 | 引导**逐步推理** | 引导**结构化分解** |

差异服务于各自核心逻辑：ReAct 需要每步思考来支持动态调整，所以提示词强调"格式化输出"；Plan-and-Solve 需要一次性生成完整计划，所以提示词强调"列表格式"。

**二、角色设定对行为的影响**

| 角色设定 | 反思焦点 | 输出风格 |
|---------|---------|---------|
| "极其严格的代码评审专家" | 算法效率、时间复杂度 | 尖锐、直接指出问题 |
| "注重可读性的开源维护者" | 代码风格、命名规范、文档 | 温和、建设性建议 |
| "安全审计专家" | 安全漏洞、边界情况 | 保守、强调防御性编程 |

结论：角色设定直接决定反思的维度和深度。选择角色时应匹配任务需求。

**三、Few-shot 示例效果**

在 ReAct 提示词中添加示例：

```
示例：
Question: 北京今天天气怎么样？
Thought: 用户想知道北京今天的天气，我需要使用搜索工具查询。
Action: Search[北京今天天气]
Observation: 北京今天晴，气温28°C。
Thought: 我已经获得了天气信息，可以回答用户了。
Action: Finish[北京今天天气晴朗，气温28°C。]
```

效果对比：
- 无示例：模型可能输出"我想想..."而非标准 Thought 格式
- 有示例：模型严格遵循 Thought-Action 格式，解析成功率显著提升

### 题目七：客服智能体设计

某电商初创公司希望使用"客服智能体"来代替真人客服，需要具备以下功能：

a. 理解用户的退款申请理由
b. 查询用户的订单信息和物流状态
c. 根据公司政策智能地判断是否应该批准退款
d. 生成一封得体的回复邮件并发送至用户邮箱
e. 如果判断决策存在一定争议（自我置信度低于阈值），能够进行自我反思并给出更审慎的建议

作为该产品的负责人：

- 你会选择本章的哪种范式（或哪些范式的组合）作为系统的核心架构？
- 这个系统需要哪些工具？请列出至少 3 个工具及其功能描述。
- 如何设计提示词来确保智能体的决策既符合公司利益，又能保持对用户的友好态度？
- 这个产品上线后可能面临哪些风险和挑战？如何通过技术手段来降低这些风险？

**参考答案：**

**一、核心架构选择**

使用 **Plan-and-Solve + ReAct + Reflection 三层混合架构**：

| 阶段 | 范式 | 作用 |
|------|------|------|
| 意图理解 | Plan-and-Solve | 分析退款请求，生成处理计划（查询订单 → 判断政策 → 生成回复） |
| 信息收集 | ReAct | 逐步调用工具查询订单、物流、政策等信息 |
| 决策审核 | Reflection | 对争议案例进行自我反思，给出审慎建议 |

**二、所需工具（至少 3 个）**

| 工具 | 功能描述 |
|------|---------|
| `query_order` | 查询用户订单信息（商品、金额、下单时间、状态） |
| `query_logistics` | 查询物流状态（发货、在途、签收、异常） |
| `check_policy` | 根据公司退款政策判断是否符合条件 |
| `send_email` | 生成并发送回复邮件到用户邮箱 |
| `calculate_confidence` | 评估决策置信度，低于阈值时触发反思 |

**三、提示词设计**

```
你是一位专业的客服代表。请遵循以下原则：
1. 始终保持礼貌和同理心，理解用户的不满
2. 严格依据公司政策做出判断，不擅自承诺
3. 如果政策不允许退款，清晰解释原因并提供替代方案
4. 涉及金额 > 500 元或用户情绪激动时，标记为"需人工审核"

公司政策摘要：
- 7天内无理由退款（特殊商品除外）
- 质量问题30天内可退
- 已使用商品不支持退款（质量问题除外）
```

**四、风险与应对**

| 风险 | 说明 | 应对措施 |
|------|------|---------|
| 决策错误 | LLM 误判退款条件，造成资金损失 | 高金额案例强制人工审核；设置置信度阈值 |
| 幻觉问题 | 捏造不存在的政策条款 | RAG 引入真实政策文档；约束"只基于提供的政策回答" |
| 用户对抗 | 用户情绪激动，AI 无法安抚 | 检测情绪关键词，自动转人工客服 |
| 隐私泄露 | 处理用户个人信息时泄露 | 工具调用时脱敏处理；日志中不记录敏感信息 |
| 攻击利用 | 用户通过精心构造的输入诱导 AI 批准不合理退款 | 输入过滤；决策结果需二次确认 |
| 系统故障 | 工具调用失败导致流程中断 | 重试机制；降级为人工处理 |
