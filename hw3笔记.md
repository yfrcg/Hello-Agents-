## 3.1 语言模型与 Transformer 架构

### 3.1.1 从 N-gram 到 RNN

#### （1）统计语言模型与 N-gram

**语言模型（LM）**是 NLP 的核心，基本任务是计算一个词序列（句子）的概率。好的语言模型能告诉我们哪些句子是流畅自然的。

句子 $S$ 由词 $w_1, w_2, \cdots, w_m$ 组成，其概率遵循**概率链式法则**：

$$P(S) = P(w_1) \cdot P(w_2|w_1) \cdot P(w_3|w_1, w_2) \cdots P(w_m|w_1, \ldots, w_{m-1})$$

直接计算几乎不可能，因此引入**马尔可夫假设**：一个词的概率仅依赖于前 $n-1$ 个词。基于此假设的模型称为 **N-gram 模型**。

- **Bigram（N=2）**：$P(w_i|w_1, \ldots, w_{i-1}) \approx P(w_i|w_{i-1})$
- **Trigram（N=3）**：$P(w_i|w_1, \ldots, w_{i-1}) \approx P(w_i|w_{i-2}, w_{i-1})$

概率通过**最大似然估计（MLE）**计算：

$$P(w_i|w_{i-1}) = \frac{Count(w_{i-1}, w_i)}{Count(w_{i-1})}$$

**计算示例**：语料库 `"datawhale agent learns datawhale agent works"`，计算 `"datawhale agent learns"` 的概率：

1. $P(\text{datawhale}) = 2/6 \approx 0.333$
2. $P(\text{agent}|\text{datawhale}) = 2/2 = 1$
3. $P(\text{learns}|\text{agent}) = 1/2 = 0.5$
4. $P(\text{datawhale agent learns}) \approx 0.333 \times 1 \times 0.5 \approx 0.167$

**N-gram 的两个致命缺陷：**

| 缺陷 | 说明 |
|------|------|
| 数据稀疏 | 未在语料中出现过的词序列，概率估计为零 |
| 泛化能力差 | 无法理解词语间的语义相似性（如 agent 和 robot） |

#### （2）神经网络语言模型与词嵌入

2003 年 Bengio 等提出**前馈神经网络语言模型**，核心思想两步走：

1. **构建语义空间**：将每个词映射到高维连续向量空间中的一个点，称为**词嵌入（Word Embedding）**。语义相似的词在空间中距离更近。
2. **学习从上下词的映射**：用神经网络学习从前 $n-1$ 个词向量到下一个词概率分布的映射。

**余弦相似度**衡量向量关系：

$$\text{similarity}(\vec{a}, \vec{b}) = \frac{\vec{a} \cdot \vec{b}}{|\vec{a}| |\vec{b}|}$$

经典示例：`vector('King') - vector('Man') + vector('Woman') ≈ vector('Queen')`

神经网络 LM 解决了泛化问题，但保留了固定上下文窗口的限制。

#### （3）RNN 与 LSTM

**循环神经网络（RNN）**引入**隐藏状态**（短期记忆），每个时间步读取当前输入，结合上一步隐藏状态生成新的隐藏状态，实现信息的序列传播。

但标准 RNN 存在**长期依赖问题**：反向传播时梯度经序列长度反复连乘，导致**梯度消失**或**梯度爆炸**。

**长短期记忆网络（LSTM）**通过引入**细胞状态**和**门控机制**解决此问题：

| 门 | 作用 |
|----|------|
| 遗忘门 | 决定从上一细胞状态中丢弃什么信息 |
| 输入门 | 决定当前输入中哪些新信息存入细胞状态 |
| 输出门 | 决定从细胞状态中输出什么信息 |

### 3.1.2 Transformer 架构详解

RNN/LSTM 顺序处理数据，第 t 步计算必须等第 t-1 步完成，无法大规模并行。2017 年 Google 提出 **Transformer**，完全抛弃循环，依赖**注意力机制**。

#### （1）编码器-解码器整体结构

- **编码器（Encoder）**："理解"整个输入句子，为每个 token 生成富含上下文信息的向量表示
- **解码器（Decoder）**："生成"目标句子，参考已生成内容和编码器的理解

#### （2）从自注意力到多头注意力

**自注意力（Self-Attention）**让模型在处理每个词时能关注到所有其他词，分配不同的"注意力权重"。

每个输入 token 获得三个角色：
- **Query（Q）**：当前 token 主动查询其他 token
- **Key（K）**：可供查询的"标签"
- **Value（V）**：token 携带的"内容"

核心公式：

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

**多头注意力（Multi-Head Attention）**将 Q、K、V 分成 h 组，每组独立计算注意力后拼接——如同 h 个不同"专家"从不同角度审视句子。

#### （3）位置前馈网络（FFN）

每个多头注意力子层之后跟随一个 FFN，对每个 token 向量独立进行两次线性变换 + ReLU：

$$\text{FFN}(x) = \max(0, xW_1 + b_1)W_2 + b_2$$

通常 `d_ff = 4 * d_model`。

#### （4）残差连接与层归一化

所有子模块被"Add & Norm"包裹：
- **残差连接（Add）**：$\text{Output} = x + \text{Sublayer}(x)$，解决梯度消失
- **层归一化（Norm）**：将单个样本的所有特征归一化为零均值单位方差，加速收敛

#### （5）位置编码

自注意力不包含位置信息。"agent learns"和"learns agent"对自注意力来说是一样的。**位置编码**通过正弦余弦函数为每个 token 添加位置向量：

$$PE_{(pos, 2i)} = \sin\left(\frac{pos}{10000^{2i/d_{\text{model}}}}\right), \quad PE_{(pos, 2i+1)} = \cos\left(\frac{pos}{10000^{2i/d_{\text{model}}}}\right)$$

**正余弦位置编码 vs RoPE 旋转位置编码**

传统正余弦位置编码（原始 Transformer）和 **RoPE（Rotary Position Embedding）**（Llama、Qwen 等现代 LLM 使用）是两种主流方案：

| 维度 | 正余弦位置编码 | RoPE 旋转位置编码 |
|------|--------------|------------------|
| 编码方式 | 将位置向量**加到** token 嵌入上 | 通过旋转矩阵**乘到** Q 和 K 向量上 |
| 编码类型 | 绝对位置（每个位置一个固定向量） | 相对位置（Q·K 的点积天然编码了两 token 的距离） |
| 计算位置 | 在嵌入层一次性添加 | 在每一层注意力计算时动态应用 |
| 长度泛化 | 超出训练长度后性能下降 | 天然支持相对位置，泛化能力更强 |
| 扩展性 | 难以扩展上下文窗口 | 可通过 NTK-aware Scaling、YaRN 等方法扩展 |

**RoPE 的核心思想**：不直接告诉模型"这个词在第 5 个位置"，而是在计算注意力时，通过旋转 Q 和 K 向量，让 `Q·K` 的点积结果只取决于两个 token 的**相对距离**。距离越近，注意力权重越高；距离越远，权重越低。

**为什么需要动态位置编码（位置插值）？**

现代 LLM 需要处理越来越长的上下文（从 2K 到 128K 甚至更长），但模型训练时的上下文窗口是固定的。当推理时序列长度超过训练长度：

| 问题 | 说明 |
|------|------|
| 正余弦编码 | 超出训练长度的位置编码是未见过的，模型无法正确理解 |
| 静态 RoPE | 训练时只见过 4K 长度，推理时 8K 的相对距离分布与训练不一致 |

**动态位置插值**的解决方案：
- **线性插值**：将长序列的位置坐标压缩到训练范围内（如 8K 位置映射到 4K），但会损失分辨率
- **NTK-aware Scaling**：调整 RoPE 的频率基数，让高频分量保持局部精度，低频分量覆盖更长距离
- **YaRN**：分频段处理，不同频率使用不同的缩放策略，兼顾局部精度和全局覆盖

> 核心思路：位置编码从"静态加法"演进到"动态旋转"，本质上是为了让模型能处理越来越长的上下文，同时保持对近距离关系的精确建模。

### 3.1.3 Decoder-Only 架构

OpenAI 的 **GPT** 提出更简单的思路：语言的核心任务就是预测下一个最可能的词。大胆**丢弃编码器，只保留解码器**——这就是 **Decoder-Only** 架构的由来。

**自回归模式**：给定起始文本 → 预测下一个词 → 拼接到输入 → 重复直到完成。

**掩码自注意力（Masked Self-Attention）**确保模型不会"偷看"未来 token：将当前位置之后的注意力分数设为极大负数，Softmax 后概率为零。

**Decoder-Only 的优势：**

| 优势 | 说明 |
|------|------|
| 统一训练目标 | "预测下一个词"的单一目标非常适合大规模无标注文本预训练 |
| 结构简单，易扩展 | 组件更少，更容易扩展到万亿参数 |
| 天然适合生成 | 自回归模式完美适配对话、写作、代码生成等生成任务 |

## 3.2 与大语言模型交互

### 3.2.1 提示工程

**提示（Prompt）**是与 LLM 通信的语言。提示工程研究如何设计精确的提示引导模型产出期望结果。

#### （1）模型采样参数

| 参数 | 作用 | 范围建议 |
|------|------|---------|
| **Temperature** | 控制输出"随机性"。T 越小越确定，T 越大越多样 | 事实任务 T<0.3，日常对话 0.3-0.7，创意任务 0.7-2 |
| **Top-k** | 保留概率最高的 k 个 token | k=1 退化为贪心采样 |
| **Top-p** | 累积概率达到 p 的 token 集合（核采样） | 动态适应分布长尾 |

优先级：Temperature → Top-k → Top-p。通常 Top-k 和 Top-p 二选一。

**Temperature 的本质与作用机制**

Temperature 是最容易被误解的参数。它不是"加随机性"，而是**调整概率分布的"尖锐程度"**。

模型最后一层输出的是 **logits**（每个 token 的原始分数 $z_i$），经 Softmax 转换为概率分布：

$$p_i = \frac{e^{z_i}}{\sum_{j=1}^{k} e^{z_j}}$$

Temperature 在 Softmax **之前**对 logits 做缩放：

$$p_i^{(T)} = \frac{e^{z_i / T}}{\sum_{j=1}^{k} e^{z_j / T}}$$

**T 的作用：**

| T 值 | 对 logits 的影响 | 概率分布 | 输出行为 |
|------|-----------------|---------|---------|
| T → 0 | $z_i / T$ 趋近无穷，差距被极度放大 | 极度尖锐，概率集中在最高 logit 上 | 确定性输出，几乎等同于贪心解码 |
| T = 1 | 不做缩放 | 原始训练时的分布 | 模型"本来"的行为 |
| T > 1 | $z_i / T$ 差距被缩小 | 变得平坦，低概率 token 获得更多机会 | 更多样、更有创意，但可能不连贯 |

**直觉理解**：想象一场考试，logits 是原始分数。T 越小，"尖子生"（最高分 token）被选中的概率越大；T 越大，"中等生"（中等分 token）也有机会被选中。

**数值示例**：

假设模型对下一个词的 logits 为：`["好": 5, "棒": 4, "不错": 3, "差": -2]`

| T | P("好") | P("棒") | P("不错") | P("差") |
|---|---------|---------|----------|---------|
| 0.1 | 99.99% | 0.01% | ≈0% | ≈0% |
| 0.7 | 61.2% | 27.5% | 11.3% | ≈0% |
| 1.0 | 47.5% | 31.7% | 20.7% | 0.1% |
| 1.5 | 38.3% | 30.6% | 24.5% | 6.6% |

可以看到：T=0.1 时几乎只选"好"；T=1.5 时"差"也有 6.6% 的概率被选中。

> **实践建议**：需要精确答案时用 T=0（或极小值）；需要自然对话时 T=0.7；需要创意时 T=1.0-1.5。不要超过 2.0，否则输出会变得毫无逻辑。

#### （2）Zero-shot / One-shot / Few-shot

| 方式 | 说明 | 示例场景 |
|------|------|---------|
| Zero-shot | 无示例，依赖预训练泛化 | 简单分类任务 |
| One-shot | 提供一个完整示例 | 格式要求明确的任务 |
| Few-shot | 提供多个覆盖不同情况的示例 | 复杂、有细微差别的任务 |

#### （3）指令微调的影响

早期 GPT（如 GPT-3）主要是"文本补全"模型。**指令微调（Instruction Tuning）**使用大量"指令-响应"格式数据进一步训练，使模型能更好地理解和遵循用户指令。日常使用的 ChatGPT、DeepSeek、Qwen 都是指令微调后的变体。

#### （4）基本提示技巧

- **角色扮演**：赋予特定角色引导回答风格、语气和知识范围
- **上下文示例**：在提示中提供清晰的输入输出示例"教"模型

#### （5）思维链（Chain-of-Thought, CoT）

对需要逻辑推理、计算或多步思考的复杂问题，在提示中加入"请逐步思考"引导模型输出逐步推理过程。

```
一个篮球队在80场比赛中赢了60%。接下来15场赢了12场。总胜率是多少？
请一步一步地思考并解答。
```

模型输出：第一赛季赢 48 场 → 总 95 场赢 60 场 → 胜率 63.16%。

### 3.2.2 文本分词

计算机只理解数字。将文本序列转换为数字序列的过程叫**分词（Tokenization）**，由**分词器（Tokenizer）**完成。

| 分词方式 | 优点 | 缺点 |
|---------|------|------|
| 基于词 | 语义完整 | 词表爆炸、OOV 问题 |
| 基于字符 | 词表小、无 OOV | 单字符缺乏语义，学习效率低 |
| **子词（Subword）** | 兼顾常见词完整性和罕见词拆分 | 现代 LLM 普遍采用 |

#### BPE 算法

**字节对编码（BPE）**是最主流的子词分词算法之一，GPT 系列使用。核心思想是"贪心"合并：

1. **初始化**：词表设为语料中出现的所有基本字符
2. **迭代合并**：统计所有相邻 token 对频率，将最高频对合并为新 token 加入词表
3. **重复**直到词表达到预设阈值

后续优化算法：
- **WordPiece**：Google BERT 使用，基于最大化语料库语言模型概率合并
- **SentencePiece**：Llama 系列使用，将空格视为普通字符，分词/解码完全可逆且语言无关

#### 分词器对开发者的意义

- **上下文窗口限制**：以 token 数而非字符数计算
- **API 成本**：按 token 数计费
- **模型行为异常**：`2+2`（无空格）可能被分为单个不常见 token，导致模型处理失败

### 3.2.3 本地调用开源 LLM

使用 **Hugging Face Transformers** 库加载和运行预训练模型：

```python
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model_id = "Qwen/Qwen1.5-0.5B-Chat"
device = "cuda" if torch.cuda.is_available() else "cpu"

tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(model_id).to(device)

messages = [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "你好，请介绍你自己。"}
]

text = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
model_inputs = tokenizer([text], return_tensors="pt").to(device)
generated_ids = model.generate(model_inputs.input_ids, max_new_tokens=512)
generated_ids = [output_ids[len(input_ids):] for input_ids, output_ids in zip(model_inputs.input_ids, generated_ids)]
response = tokenizer.batch_decode(generated_ids, skip_special_tokens=True)[0]
print(response)
```

### 3.2.4 模型选择

选择语言模型是在性能、成本、速度和部署方式之间的权衡。

**关键考虑因素：**

| 因素 | 说明 |
|------|------|
| 性能与能力 | 不同模型擅长不同任务（推理/代码/创意/多语言） |
| 成本 | 闭源按 token 付费，开源需硬件和运维成本 |
| 速度（延迟） | 实时交互场景需要低延迟模型 |
| 上下文窗口 | 长文档理解或长期对话记忆需要大窗口（128K+） |
| 部署方式 | API 最简但数据外传，本地部署保隐私但需技术能力 |
| 生态与工具链 | 热门模型有更丰富的社区支持和兼容框架 |
| 可微调性 | 领域特定 Agent 需要自定义微调能力 |
| 安全与伦理 | 偏见、毒性、幻觉表现 |

**闭源模型概览：**

| 模型 | 特点 |
|------|------|
| OpenAI GPT 系列 | GPT-3 → ChatGPT（RLHF）→ GPT-4（多模态）→ GPT-5 |
| Google Gemini 系列 | 原生多模态，Gemini 2.5 增强推理和上下文窗口 |
| Anthropic Claude 系列 | AI 安全优先，擅长长文档，Claude 4 Opus 推进通用智能 |
| 国内主流 | 百度文心、腾讯混元、华为盘古、讯飞星火、月之暗面 |

**开源模型概览：**

| 模型 | 特点 |
|------|------|
| Meta Llama 系列 | Llama 4 首次采用 MoE 架构，Scout/Maverick/Behemoth 三款 |
| Mistral AI 系列 | 法国公司，"小尺寸高性能"，Mistral Medium 3.1 超越多个竞品 |
| 国内开源 | 阿里 **Qwen** 系列、清华/智谱 **ChatGLM** 系列，中文能力强 |

## 3.3 缩放定律与 LLM 的局限性

### 3.3.1 缩放定律

**缩放定律（Scaling Laws）**揭示了模型性能与参数量、训练数据量、计算资源之间的幂律关系。在对数坐标下，三者与性能呈平滑幂律曲线。

**Chinchilla 定律**（DeepMind 2022）：给定计算预算下，**参数量与训练数据量之间存在最优比例**。最优模型应该比之前认为的更小，但用更多数据训练。70B 参数的 Chinchilla 用 4 倍数据超越了 175B 的 GPT-3。

**能力涌现（Emergence）**：当模型规模达到某个阈值时，突然展现出小模型完全不具备的新能力——思维链推理、指令遵循、多步推理、代码生成等在千亿/万亿参数级别才显著出现。

### 3.3.2 模型幻觉

**模型幻觉（Hallucination）**指 LLM 生成与客观事实、用户输入或上下文矛盾的内容，或捏造不存在的事实、实体、事件。

| 类型 | 说明 |
|------|------|
| 事实性幻觉 | 与现实世界事实不一致 |
| 忠实性幻觉 | 未忠实反映源文本含义（摘要、翻译中） |
| 内在幻觉 | 直接与输入信息矛盾 |

**原因**：训练数据含错误/矛盾；自回归机制只预测下一个最可能 token，无内建事实验证；复杂推理链中的误差累积。

**改善可靠性方法：**

| 层级 | 方法 |
|------|------|
| 数据层 | 高质量数据清洗、引入事实知识、RLHF |
| 模型层 | 探索新架构、让模型表达不确定性 |
| 推理层 | **RAG**（检索增强生成）、多步推理与验证、引入外部工具（搜索引擎、计算器、代码解释器） |

## 课后练习

> **提示**：以下部分习题没有标准答案，旨在帮助学习者深入理解大语言模型的核心原理和实践要点。

### 题目一：语言模型演进

- 使用本章提供的迷你语料库 `"datawhale agent learns datawhale agent works"`，用 Bigram 模型计算 `"agent works"` 的概率
- 解释马尔可夫假设的含义以及 N-gram 模型的根本局限
- 说明神经网络语言模型（RNN/LSTM）和 Transformer 是如何克服 N-gram 局限的，各自有什么优势

**参考答案：**

**一、Bigram 计算 P("agent works")**

语料库：`"datawhale agent learns datawhale agent works"`，共 6 个 token。

1. $P(\text{agent}) = 2/6 \approx 0.333$
2. $P(\text{works}|\text{agent}) = Count(\text{agent, works}) / Count(\text{agent}) = 1/2 = 0.5$
3. $P(\text{agent works}) \approx 0.333 \times 0.5 \approx 0.167$

**二、马尔可夫假设与 N-gram 局限**

马尔可夫假设：一个词的概率仅依赖于前 $n-1$ 个词，而非整个历史。这大大简化了计算，但也带来了根本局限：

| 局限 | 说明 |
|------|------|
| 数据稀疏 | 未在语料中出现过的词序列概率为零，即使语义合理 |
| 泛化能力差 | 无法理解词语间的语义相似性（agent 和 robot 被视为完全不同的符号） |
| 上下文窗口固定 | N 越大越精确，但数据稀疏问题越严重，通常 N≤5 |

**三、如何克服 N-gram 局限**

| 模型 | 克服方式 | 优势 |
|------|---------|------|
| 神经网络 LM | 将词映射为连续向量（词嵌入），语义相似的词向量距离近，解决泛化问题 | 可学习语义关系，agent 的经验可迁移到 robot |
| RNN/LSTM | 引入隐藏状态，理论上可处理任意长度序列，突破固定窗口限制 | 能捕捉长距离依赖，LSTM 的门控机制缓解梯度消失 |
| Transformer | 自注意力机制直接建模任意两个位置间的关系，支持并行计算 | 并行训练效率高，完美捕获长距离依赖，可扩展到超大规模 |

### 题目二：Transformer 架构

- Self-Attention 的核心思想是什么？
- 为什么 Transformer 可以并行处理序列而 RNN 必须串行？位置编码在其中起到了什么作用？
- Decoder-Only 与完整的 Encoder-Decoder 有什么区别？为什么主流 LLM 都采用 Decoder-Only？

**参考答案：**

**一、Self-Attention 核心思想**

让模型在处理每个词时能关注到序列中所有其他词，并分配不同的"注意力权重"。通过 Query、Key、Value 三个角色，计算每对词之间的相关性，加权聚合信息。

类比：阅读"The agent learns because **it** is intelligent"时，读到"it"大脑会更关注"agent"来消解指代。Self-Attention 数学化地建模了这一现象。

**二、并行 vs 串行，位置编码的作用**

| 维度 | RNN | Transformer |
|------|-----|-------------|
| 计算方式 | 必须按顺序逐步计算，第 t 步依赖第 t-1 步 | 所有位置的注意力可同时计算（矩阵乘法） |
| 并行性 | 无法并行 | 完全并行，充分利用 GPU |
| 长距离依赖 | 信息经过多步传递会衰减 | 任意两个位置直接交互，无衰减 |

位置编码的作用：自注意力本身不包含位置信息（"agent learns"和"learns agent"对自注意力来说一样）。位置编码通过正弦余弦函数为每个 token 添加唯一的位置向量，让模型知道词的顺序。

**三、Decoder-Only vs Encoder-Decoder**

| 维度 | Encoder-Decoder | Decoder-Only |
|------|----------------|--------------|
| 结构 | 编码器理解输入 + 解码器生成输出 | 只有解码器，自回归生成 |
| 适合任务 | 翻译、摘要等"输入→输出"任务 | 通用生成（对话、写作、代码） |
| 训练目标 | 编码和解码两个目标 | 统一的"预测下一个词"目标 |
| 复杂度 | 组件多，扩展难度大 | 结构简单，易于扩展到万亿参数 |

主流 LLM 采用 Decoder-Only 的原因：
- "预测下一个词"的统一目标非常适合大规模无标注文本预训练
- 结构简单，更容易扩展（GPT-4、Llama 都是万亿参数级别）
- 自回归模式天然适合所有生成任务，是构建通用 Agent 的理想基础

### 题目三：子词分词

为什么不能直接用字符或词作为模型的输入单元？BPE 解决了什么问题？

**参考答案：**

**为什么不能用词？**

| 问题 | 说明 |
|------|------|
| 词表爆炸 | 英语有数十万词，中文更甚，词表过大导致模型参数膨胀 |
| OOV 问题 | 未见过的新词（如专有名词、拼写错误）无法处理 |
| 形态学割裂 | "look"、"looks"、"looking"被视为三个完全不同的 token，无法共享语义 |

**为什么不能用字符？**

| 问题 | 说明 |
|------|------|
| 语义缺失 | 单个字符几乎没有独立语义，学习效率极低 |
| 序列过长 | 同样文本的字符序列远长于词序列，增加计算负担 |

**BPE 解决了什么问题？**

BPE 在词和字符之间找到平衡：
- 常见词保持完整（如 "the"、"agent"）
- 罕见词拆分为有意义的子词（如 "Tokenization" → "Token" + "ization"）
- 词表大小可控（通常 32K-100K）
- 无 OOV 问题（任何新词都可拆分为已知子词的组合）

### 题目四：实践部署练习

- 在本地部署一个轻量级开源模型（推荐 Qwen3-0.6B），调整采样参数并观察输出效果
- 针对某个特定任务，设计并比较不同的提示策略（Zero-shot、Few-shot、Chain-of-Thought）
- 从性能、成本、可控性、隐私等维度对比闭源与开源模型
- 如果要为企业搭建一个客服智能体，你会选择哪种类型的模型？需要考虑哪些因素？

**参考答案：**

**一、采样参数实验观察**

| 参数设置 | 观察效果 |
|---------|---------|
| Temperature=0, Top-k=1 | 输出确定、重复，适合事实性问答 |
| Temperature=0.7, Top-p=0.9 | 输出自然、多样，适合日常对话 |
| Temperature=1.5 | 输出创意但可能不连贯，适合头脑风暴 |

**二、提示策略对比（以情感分析任务为例）**

| 策略 | 示例 | 效果 |
|------|------|------|
| Zero-shot | "判断以下评论的情感：'这家店太棒了'" | 简单任务可用，复杂任务不稳定 |
| Few-shot | 提供 3 个正/负/中性示例后再问 | 显著提升格式一致性和准确率 |
| CoT | "请逐步分析：1.提取情感词 2.判断极性 3.给出结论" | 复杂推理任务效果最好，但 token 消耗更多 |

**三、闭源 vs 开源对比**

| 维度 | 闭源模型 | 开源模型 |
|------|---------|---------|
| 性能 | 通常领先（GPT-4、Claude Opus） | 近年追赶迅速（Llama 4、Qwen 3） |
| 成本 | 按 token 付费，大规模使用成本高 | 一次性硬件成本，长期使用更划算 |
| 可控性 | 受限于 API 参数 | 可完全控制模型行为、可微调 |
| 隐私 | 数据发送到第三方 | 数据完全在本地，零泄露风险 |
| 维护 | 无需维护 | 需要 GPU 资源和运维能力 |

**四、企业客服智能体选型建议**

推荐**混合方案**：
- 主力模型：闭源 API（如 GPT-4o-mini、Claude Haiku）处理常规问题，成本低、质量稳定
- 敏感场景：本地部署开源模型处理涉及隐私的数据
- 需考虑因素：响应延迟（<2s）、并发能力、成本预算、数据合规、可微调性（适配业务知识）

### 题目五：模型幻觉

- 选择一种缓解方法，解释其工作原理和适用场景
- 调研其他前沿的幻觉缓解方法及其改进效果

**参考答案：**

**一、RAG（检索增强生成）**

**工作原理：**

```
用户提问
    ↓
从外部知识库（文档、数据库、网页）中检索与问题相关的内容
    ↓
将检索结果作为上下文注入 Prompt
    ↓
LLM 基于检索到的事实信息生成回答
    ↓
输出有据可查的回答
```

**适用场景：**
- 企业知识库问答（回答基于公司文档，而非模型记忆）
- 医疗/法律等需要准确引用的领域
- 需要实时信息的场景（新闻、股价）

**优势**：回答有据可查，可追溯来源；知识可热更新，无需重新训练模型。

**二、其他前沿方法**

| 方法 | 原理 | 改进效果 |
|------|------|---------|
| **自一致性（Self-Consistency）** | 对同一问题生成多个推理路径，投票选最一致的答案 | 数学推理准确率提升 10-20% |
| **工具调用（Tool Use）** | 让模型调用计算器、搜索引擎等外部工具获取精确结果 | 事实性错误显著减少 |
| **RLHF/RLAIF** | 用人类或 AI 反馈训练模型减少幻觉输出 | 从训练层面降低幻觉倾向 |
| **思维链验证（CoT Verification）** | 生成推理链后，让模型自我检查每一步是否正确 | 发现并修正推理中的错误 |
| **置信度校准** | 训练模型表达"我不确定"，低置信度时拒绝回答 | 减少"自信地胡说"的情况 |

### 题目六：论文阅读助手设计

设计一个"论文阅读助手"智能体，能总结论文、回答问题、提取关键信息、对比多篇论文观点：

- 你会选择什么基础模型？需要考虑哪些因素？
- 如何设计提示来理解学术论文？如何处理超过上下文窗口限制的论文？
- 如何确保生成的信息准确、客观、忠实于原文？

**参考答案：**

**一、基础模型选择**

推荐 **Claude（长上下文版本）或 GPT-4o**，考虑因素：

| 因素 | 说明 |
|------|------|
| 上下文窗口 | 论文通常 10-30K token，需要至少 128K 窗口以支持多篇对比 |
| 长文档理解能力 | Claude 系列在长文档理解和忠实性方面表现突出 |
| 推理能力 | 学术论文涉及复杂逻辑推理和数学公式 |
| 多语言 | 需要处理中英文论文 |

**二、提示设计与长论文处理**

**提示设计：**

```
你是一位学术论文阅读助手。请严格按照以下要求工作：
1. 总结时必须引用原文关键句子
2. 回答问题时标注信息来源（页码/段落）
3. 对不确定的内容明确标注"原文未提及"
4. 对比多篇论文时，列出支持和反对的证据
```

**长论文处理（超过上下文窗口时）：**

- **分段摘要**：将论文按章节拆分，分别摘要后合并
- **RAG 检索**：将论文向量化存入知识库，按问题检索相关段落
- **层级阅读**：先读摘要和结论，再按需深入具体章节

**三、确保信息准确性**

| 策略 | 说明 |
|------|------|
| 引用溯源 | 每个回答必须附带原文引用（页码、段落号） |
| 忠实性约束 | 在提示中明确要求"只基于提供的文本回答，不要添加外部知识" |
| 不确定性表达 | 训练模型对不确定内容说"原文未明确说明" |
| 交叉验证 | 对关键结论，要求模型从论文不同部分寻找支持/矛盾证据 |
| 人工审核 | 高风险场景（如引用用于论文写作）加入人工审核环节 |
